Microsoft Security Bulletin MS09-013 - Crítica
==============================================

Vulnerabilidades no Windows HTTP Services podem permitir a execução remota de código (960803)
---------------------------------------------------------------------------------------------

Publicado: terça-feira, 14 de abril de 2009 | Atualizado: quarta-feira, 29 de abril de 2009

**Versão:** 1.1

### Informações Gerais

#### Sinopse

Esta atualização de segurança resolve uma vulnerabilidade divulgada publicamente e duas reportadas em particular no Microsoft Windows HTTP Services (WinHTTP). A vulnerabilidade mais grave pode permitir a execução remota de código. O invasor que explorar com êxito essa vulnerabilidade poderá assumir o controle total de um sistema afetado. O invasor poderá instalar programas; exibir, alterar ou excluir dados; ou criar novas contas com direitos totais de usuário. Os usuários cujas contas são configuradas com poucos direitos de usuário no sistema correm menos riscos do que aqueles que possuem direitos administrativos.

Esta atualização de segurança é classificada como crítica para todas as edições do Microsoft Windows 2000, Windows XP, Windows Server 2003, Windows Vista e Windows Server 2008 com suporte. Para obter mais informações, consulte a subseção **Software afetado e não afetado**, nesta seção.

A atualização de segurança elimina as vulnerabilidades mudando a maneira como o Windows HTTP Services lida com erros e valida certificados, e garantindo que o Windows HTTP Services use corretamente mecanismos de proteção de reflexão de credencial NTLM. Para obter mais informação sobre as vulnerabilidades, consulte a subseção Perguntas frequentes relacionada à entrada da vulnerabilidade específica presente na próxima seção, **Informações sobre a vulnerabilidade**.

**Recomendação.** A maioria dos clientes tem o recurso de atualizações automáticas habilitado e não precisará tomar nenhuma providência porque esta atualização de segurança será baixada e instalada automaticamente. Os clientes que não têm o recurso de atualizações automáticas habilitado precisarão verificar as atualizações e instalar esta atualização manualmente. Para obter informações sobre opções de configuração específicas em atualizações automáticas, consulte o [Artigo 294871 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/294871).

Para instalações feitas por administradores ou empresas, ou por usuários finais que desejam instalar esta atualização de segurança manualmente, a Microsoft recomenda que os clientes apliquem a atualização imediatamente usando software de gerenciamento de atualização ou verificando se há atualizações no serviço [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Consulte também a seção **Orientação e ferramentas de detecção e implantação** mais adiante neste boletim.

**Problemas conhecidos.** O [Artigo 960803 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/960803) documenta os problemas atualmente conhecidos que os clientes podem enfrentar ao instalar esta atualização de segurança. O artigo também documenta as soluções recomendadas para esses problemas.

**Onde estão os detalhes das informações sobre o arquivo?**    
Informações detalhadas sobre o arquivo podem ser encontradas no [Artigo 960803 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/960803).

Sim. Essas vulnerabilidades foram relatadas após o lançamento do Windows Server 2008 Service Pack 2 Beta, Windows Vista Service Pack 2 Beta e Windows 7 Beta. Recomendamos que os clientes executando essas plataformas baixem e apliquem a atualização a seus sistemas.

As atualizações de segurança estão disponíveis no [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) e no [Windows Update](http://go.microsoft.com/fwlink/?linkid=21130).

**Como esta atualização de segurança se relaciona com o boletim MS09-014?**    
A Vulnerabilidade de reflexão de credencial do Windows HTTP Services, [CVE-2009-0550](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0550), descrita neste boletim de segurança, também afeta o Internet Explorer e a API de Internet do Windows (WinINet). A Microsoft recomenda aos usuários do Internet Explorer que também instalem a atualização de segurança fornecida no [MS09-014](http://technet.microsoft.com/security/bulletin/ms09-014), a Atualização de segurança cumulativa para Internet Explorer (963027).

**Quais são os problemas conhecidos que os clientes podem enfrentar ao instalar esta atualização de segurança?**    
O [Artigo 960803 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/960803) documenta os problemas atualmente conhecidos que os clientes podem enfrentar ao instalar esta atualização de segurança. O artigo também documenta as soluções recomendadas para esses problemas.

**Por que esta atualização elimina várias vulnerabilidades de segurança reportadas?**    
Esta atualização contém suporte para diversas vulnerabilidades porque as modificações necessárias para corrigir esses problemas foram feitas em arquivos relacionados. Em vez de instalar várias atualizações quase iguais, os clientes poderão instalar apenas esta atualização

**Esta atualização contém alterações relacionadas à segurança da funcionalidade?**    
Sim, para resolver o problema descrito em CVE-2009-0089, a Microsoft implementou a nova chave de Registro UseFQDNFallbackForCnCompare em SOFTWARE\\Microsoft\\Windows\\CurrentVersion\\Internet Settings\\WinHttp, que controla o comportamento do WinHTTP na validação do nome de domínio totalmente qualificado de um host com relação à entrada de nome diferenciado em um certificado.

Por padrão, não existe uma chave de Registro que garanta que o WinHTTP valide que o nome diferenciado (DN) contido no certificado seja idêntico ao DN com que a conexão foi iniciada. O comportamento antigo pode ser reativado configurando-se esta chave do Registro como verdadeira. Neste caso, a validação será menos rigorosa e aceitará se o certificado corresponder a uma alteração de nome feita por encaminhamento no nível de DNS. Esse é um comportamento menos seguro, que é padrão antes da instalação desta atualização de segurança.

Vulnerabilidade de estouro negativo de número inteiro do Windows HTTP Services - CVE-2009-0086  
----------------------------------------------------------------------------------------------
  

Existe uma vulnerabilidade de execução remota de código na maneira como o Windows HTTP Services manipula valores específicos que são retornados por um servidor Web remoto. O invasor que explorar com êxito essa vulnerabilidade poderá assumir o controle total de um sistema afetado. Um invasor poderá então instalar programas; exibir, alterar ou excluir dados; ou criar novas contas com os mesmos direitos de usuário do serviço ou aplicativo que chama a API do WinHPPT para conectar-se ao servidor Web do invasor.
  
Para exibir essa vulnerabilidade como uma entrada padrão na lista Common Vulnerabilities and Exposures, consulte [CVE-2009-0086](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0086).
  
#### Fatores atenuantes da Vulnerabilidade de estouro negativo de número inteiro do Windows HTTP Services - CVE-2009-0086
  
A atenuação refere-se a uma configuração, configuração comum ou prática recomendada geral, existindo num estado padrão, que possa reduzir a gravidade de exploração de uma vulnerabilidade. Os fatores de atenuação a seguir podem ser úteis em sua situação:
  
-   Nenhuma edição com suporte do Internet Explorer é afetada por esta vulnerabilidade porque o Windows HTTP Services não é usado pelo software.
  
#### Soluções alternativas para a Vulnerabilidade de estouro negativo de número inteiro do Windows HTTP Services - CVE-2009-0086
  
A Microsoft não identificou nenhuma solução alternativa para esta vulnerabilidade.
  
#### Perguntas frequentes sobre a Vulnerabilidade de estouro negativo de número inteiro do Windows HTTP Services - CVE-2009-0086
  
**Qual é o escopo da vulnerabilidade?**      
Essa é uma vulnerabilidade de execução remota de código. O invasor que explorar com êxito essa vulnerabilidade poderá assumir o controle total de um sistema afetado. Um invasor poderá então instalar programas; exibir, alterar ou excluir dados; ou criar novas contas com os mesmos direitos de usuário do serviço ou aplicativo que chama a API do WinHPPT para conectar-se ao servidor Web do invasor.
  
Esta vulnerabilidade afeta componentes do Windows e aplicativos de terceiros que usam o Windows HTTP Services e são configurados para conexão com servidores Web remotos não confiáveis.
  
**O que provoca a vulnerabilidade?**      
A vulnerabilidade é causada pela falta de validação de um parâmetro retornado por um servidor Web remoto. Isso pode levar à execução arbitrária do código.
  
**O que é o Windows HTTP Services?**      
O Windows HTTP Services (WinHTTP) fornece aos desenvolvedores uma interface de programação de aplicativo (API) de cliente HTTP para enviar solicitações pelo protocolo de HTTP para servidores Web. O WinHTTP pode ser usado tanto por componentes do Microsoft Windows quanto por software de terceiros. O WinHTTP é, entre outros, usado pelo serviço Universal Plug and Play.
  
**O que é Universal Plug and Play?**      
O Universal Plug and Play (UPnP) é um recurso que permite aos dispositivos em uma rede descobrir outros dispositivos e determina como trabalhar com eles. Pode-se compreender o UPnP comparando-o com o recurso plug-and-play (PnP), já conhecido da maioria dos usuários do Windows. O recurso PnP permite que o sistema operacional detecte novos itens de hardware quando você os instala em um sistema. Por exemplo, quando você instala um novo mouse no computador, o PnP permite que o Windows detecte esse dispositivo, carregue os drivers necessários e comece a usar o novo mouse. O UPnP estende esse conceito a dispositivos em uma rede, em vez do sistema local. O UPnP permite que os computadores detectem outros dispositivos na rede e determina como usá-los. Por exemplo, um computador pode usar o UPnP para detectar se há impressoras que ele possa usar na rede e para aprender como usá-las.
  
**O que é o Protocolo SSDP?**      
A descoberta de dispositivos é o processo pelo qual os computadores com suporte a UPnP se tornam cientes da disponibilidade de dispositivos que podem usar e aprendem como solicitar serviços desses dispositivos. Quando um computador com suporte a UPnP é inicializado, já pode haver dispositivos na rede que ele pode usar. Para determinar se é esse o caso, o computador envia uma solicitação de difusão (chamada diretiva M-SEARCH), pedindo que todo dispositivo com suporte a UPnP na sub-rede responda diretamente e forneça informações sobre seu uso.
  
De forma semelhante, quando um dispositivo com suporte a UPnP (por exemplo, uma impressora com suporte a UPnP) é inicializado, já pode haver computadores com suporte a UPnP na rede que gostariam de usá-lo. O dispositivo transmite uma mensagem (chamada NOTIFY) a todos computadores na sub-rede, anunciando sua presença na rede e convidando os computadores a usar seus serviços.
  
**Para que um invasor pode usar a vulnerabilidade?**      
Um invasor que explore esta vulnerabilidade com êxito pode obter os mesmos direitos de usuário do serviço ou aplicativo que chama a API do WinHPPT para conectar-se ao servidor Web do invasor. Os aplicativos executados com contas configuradas com menos direitos de usuário no sistema podem ser menos afetados do que aqueles executados com contas com direitos de usuário administrativos.
  
**De que forma o invasor pode explorar a vulnerabilidade?**      
Um invasor precisaria forçar ou persuadir um programa que use o Windows HTTP Services a se conectar a seu servidor Web mal-intencionado. É importante observar que o serviço Universal Plug and Play (UPnP) usa as bibliotecas do WinHTTP. Quando esse serviço é ativado, usuários mal-intencionados na sub-rede local podem responder a solicitações SSDP e levar o serviço UPnP a se conectar usando o WinHTTP a um host mal-intencionado que pode então explorar essa vulnerabilidade. O serviço UPnP é executado na conta LocalService, que é uma conta com privilégios limitados no sistema.
  
**Quais são os sistemas que mais correm riscos com a vulnerabilidade?**      
Principalmente as estações de trabalho e os servidores de terminal estão correndo risco. Os servidores correm mais riscos se os administradores permitirem que os usuários façam logon em servidores e executem programas. Entretanto, as práticas recomendadas não recomendam esse procedimento.
  
**O que a atualização faz?**      
A atualização elimina a vulnerabilidade, modificando o modo como o Windows HTTP Services lida com o erro que resulta na condição de vulnerabilidade.

#### Perguntas frequentes sobre a Vulnerabilidade de incompatibilidade de nome em certificado do Windows HTTP Services - CVE- 2009-0089
  
**Qual é o escopo da vulnerabilidade?**      
Esta é uma vulnerabilidade de falsificação existente no Windows HTTP Services. Quando combinado com outros ataques específicos, como falsificação de DNS, isso pode permitir a um invasor falsificar o certificado digital de um site para qualquer aplicativo que use o Windows HTTP Services.
  
**O que provoca a vulnerabilidade?**      
Quando um aplicativo chama o Windows HTTP Services para configurar uma conexão a um servidor Web remoto, o WinHTTP valida somente o nome de domínio totalmente qualificado do URL com relação ao certificado do site original a que o host tenta se conectar. Combinações específicas de falsificação de DNS podem encaminhar uma conexão para um servidor Web diferente que tenha um certificado válido para esse host específico, mas não para a página da Web com a qual o WinHTTP iniciou a conexão. A combinação de falsificação de DNS com esta vulnerabilidade pode fazer com que o WinHTTP incorretamente considere o certificado do servidor Web remoto como aceitável. Como o WinHTTP depende do aplicativo para interação com o usuário, o usuário talvez não seja advertido quanto a essa discrepância pelo aplicativo.
  
**O que é o Windows HTTP Services?**      
O Windows HTTP Services (WinHTTP) fornece aos desenvolvedores uma interface de programação de aplicativo (API) de cliente HTTP para enviar solicitações pelo protocolo de HTTP para outros servidores HTTP.
  
**O que são certificados digitais?**      
O certificado digital é uma credencial comum usada para autenticar identidades. Uma organização confiável atribui um certificado a um indivíduo ou entidade, a entidade desse certificado. A organização confiável que emite o certificado é uma Autoridade de Certificação (CA), conhecida como emissora do certificado. Uma CA confiável emite um certificado somente depois de verificar a identidade da entidade do certificado.
  
**O que é falsificação de DNS?**      
O DNS (Domain Name System) é um dos conjuntos de protocolos padrão do setor que abrangem TCP/IP. O DNS é implementado usando-se dois componentes de software: o servidor de DNS e o cliente de DNS (ou resolvedor). Ambos os componentes são executados como aplicativos de serviços de segundo plano. Os recursos de rede são identificados por endereços IP numéricos, mas estes endereços IP são difíceis de lembrar para os usuários de rede. O banco de dados do DNS contém registros que mapeiam nomes alfanuméricos fáceis de usar de recursos de rede para os endereços IP usados por esses recursos para comunicação. Desta maneira, o DNS age como um dispositivo mnemônico, tornando mais fácil para os usuários da rede lembrar os recursos da rede. Para obter mais informações e ver diagramas lógicos que ilustram como o DNS se ajusta a outras tecnologias do Windows, leia o artigo da TechNet [O que é DNS?](http://technet.microsoft.com/en-us/library/cc787921.aspx) (em inglês).
  
A Falsificação de DNS é uma categoria ampla de ataques em que, em algum ponto do sistema DNS, respostas DNS incorretas são introduzidas por um terceiro que normalmente não é responsável por essas respostas. Um ataque desse tipo pode fazer com que um nome de domínio ou de host resulte indevidamente em um endereço IP, sem a intenção dos proprietários desse nome. Assim, um host pode ser levado a se conectar a um servidor de terceiros, potencialmente mal-intencionado.
  
**Para que um invasor pode usar a vulnerabilidade?**      
Um invasor pode representar um site seguro e oferecer conteúdo mal-intencionado ao aplicativo que usa o Windows HTTP Services, que confiaria nesse conteúdo como se fosse originado de um site seguro.
  
**De que forma o invasor pode explorar a vulnerabilidade?**      
Um invasor precisa combinar essa vulnerabilidade com falsificação de DNS ou com um ataque semelhante para dirigir a conexão de saída a um servidor que esteja sob seu controle. O invasor então ofereceria um certificado válido para o seu próprio site, que seria incorretamente validado pelo Windows HTTP Services como válido para o site do host de destino a que o usuário tentou se conectar.
  
**Quais são os sistemas que mais correm riscos com a vulnerabilidade?**      
Principalmente as estações de trabalho e os servidores de terminal estão correndo risco. Os servidores correm mais riscos se os administradores permitirem que os usuários façam logon em servidores e executem programas. Entretanto, as práticas recomendadas não recomendam esse procedimento.
  
**O que a atualização faz?**      
A atualização remove a vulnerabilidade mudando a maneira como o Windows HTTP Services valida certificados pertencentes a um site seguro.

Vulnerabilidade de reflexão da credencial do Windows HTTP Services - CVE-2009-0550  
----------------------------------------------------------------------------------
  

Existe uma vulnerabilidade de execução remota de código na maneira como o Windows HTTP Services lida com credenciais NTLM quando um usuário se conecta a um servidor Web do invasor. Essa vulnerabilidade permite que um invasor reproduza novamente as credenciais do usuário de volta para ele e execute o código no contexto do usuário conectado. Se um usuário tiver feito logon com direitos administrativos, o invasor que explorar com êxito essa vulnerabilidade poderá ter o controle total do sistema afetado. O invasor poderá instalar programas; exibir, alterar ou excluir dados; ou criar novas contas com direitos totais de usuário. Os usuários cujas contas são configuradas com poucos direitos de usuário no sistema correm menos riscos do que aqueles que possuem direitos administrativos.
  
Para exibir essa vulnerabilidade como uma entrada padrão na lista Common Vulnerabilities and Exposures, consulte [CVE-2009-0550](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0550).
  
#### Fatores atenuantes da Vulnerabilidade de reflexão da credencial do Windows HTTP Services - CVE-2009-0550
  
A atenuação refere-se a uma configuração, configuração comum ou prática recomendada geral, existindo num estado padrão, que possa reduzir a gravidade de exploração de uma vulnerabilidade. Os fatores de atenuação a seguir podem ser úteis em sua situação:
  
-   No cenário de ataque na Web, o invasor terá que hospedar um site contendo uma página da Web usada para explorar essa vulnerabilidade. Além disso, sites comprometidos e sites que aceitem ou hospedem conteúdo fornecido pelo usuário ou anúncios pode conter conteúdo especialmente desenvolvido que pode explorar esta vulnerabilidade. Em todos os casos, entretanto, um invasor não teria como forçar os usuários a visitarem os sites. Em vez disso, o invasor teria que convencer os usuários a visitar o site, geralmente fazendo com que eles cliquem em um link em um email ou mensagem do Instant Messenger que leva o usuário ao site do invasor.  
-   O invasor que explorar com êxito a vulnerabilidade poderá obter os mesmos direitos que o usuário local. Os usuários cujas contas são configuradas com poucos direitos de usuário no sistema correm menos riscos do que aqueles que possuem direitos administrativos.  
-   Por padrão, todas as edições com suporte do Windows Vista e do Windows Server 2008 têm o perfil do firewall de Windows definido como público, o que reduz a exposição destas plataformas a esta vulnerabilidade, visto que, no perfil público, o firewall bloqueia o tráfego do SMB. Como, para que o ataque fosse bem-sucedido e as credenciais refletidas, o cliente precisaria estar executando um serviço que aceita a autenticação NTLM, o vetor de ataque mais comum é o SMB.

Essa é uma vulnerabilidade de execução remota de código. O invasor que explorar com êxito essa vulnerabilidade poderá assumir o controle total de um sistema afetado. O invasor poderá instalar programas; exibir, alterar ou excluir dados; ou criar novas contas com direitos totais de usuário.

**De que forma o invasor pode explorar a vulnerabilidade?**

Um invasor teria que hospedar um servidor Web especialmente criado que capturasse o log NTLM de credenciais do usuário. O invasor então reproduziria essas credenciais para um serviço no cliente do usuário que aceite a autenticação NTLM. Não há como um invasor forçar usuários a visitar um site especialmente criado. **O que a atualização faz?**      
A atualização de segurança corrige a vulnerabilidade por garantir que o Windows HTTP Services aceite corretamente os mecanismos de proteção de credenciais disponível no Windows.

É possível usar os componentes do [Avaliador de compatibilidade com atualizações](http://technet2.microsoft.com/windowsvista/en/library/4279e239-37a4-44aa-aec5-4e70fe39f9de1033.mspx?mfr=true) incluídos no [Kit de ferramentas de compatibilidade de aplicativos 5.0](http://www.microsoft.com/downloads/details.aspx?familyid=24da89e9-b581-47b0-b45e-492dd6da2971&displaylang=en) para agilizar o teste e a validação de atualizações do Windows com relação aos aplicativos instalados.
  
O Kit de ferramentas de compatibilidade de aplicativos (ACT) contém as ferramentas e a documentação necessárias para avaliar e atenuar problemas de compatibilidade com aplicativos antes da implantação do Microsoft Windows Vista, de uma atualização do Windows, de uma atualização de segurança da Microsoft ou de uma nova versão do Windows Internet Explorer em seu ambiente.

**Verificando a aplicação da atualização**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma atualização de segurança foi aplicada em um sistema afetado, é possível utilizar a ferramenta MBSA (Microsoft Baseline Security Analyzer). Consulte a seção **Orientação e ferramentas de detecção e implantação** neste boletim para obter mais informações.

-   **Verificação de Versão do Arquivo**

    Como existem várias versões do Microsoft Windows, as etapas a seguir podem ser diferentes no seu sistema. Se isso acontecer, consulte a documentação do produto para completar estas etapas.

    1.  Clique em **Iniciar** e, em seguida, clique em **Pesquisar**.
    2.  No painel **Resultados da Pesquisa**, clique em **Todos os arquivos e pastas**, em **Search Companion**.
    3.  Na caixa **Todo ou parte do nome do arquivo**, digite um nome de arquivo da tabela de informações de arquivo correta e clique em **Pesquisar**.
    4.  Na lista de arquivos, clique com o botão direito do mouse em um nome de arquivo da tabela de informações de arquivo correta e clique em **Propriedades**.
        **Observação** Dependendo da edição do sistema operacional, ou dos programas que são instalados em seu sistema, alguns arquivos que estão listados na tabela de informações de arquivo podem não estar instalados.
    5.  Na guia **Versão**, determine a versão do arquivo instalado no sistema, comparando-o com a versão documentada na tabela de informações de arquivos correta.
        **Observação** Atributos que não sejam a versão do arquivo podem ser alterados durante a instalação. A comparação de outros atributos de arquivo com as informações na tabela não é um método suportado para verificar se a atualização foi aplicada. Além disso, em alguns casos, arquivos podem ser renomeados durante a instalação. Se as informações sobre o arquivo ou a versão não estiverem presentes, use um dos outros métodos disponíveis para verificar a instalação da atualização.

-   **Verificação da Chave do Registro**

    Você também pode ser capaz de verificar os arquivos que esta atualização de segurança instalou examinando as chaves do Registro na **Tabela de referência** nesta seção.

    Essas chaves do Registro podem não conter uma lista completa dos arquivos instalados. Além disso, talvez essas chaves do Registro não sejam criadas corretamente se um administrador ou um OEM integrar ou adicionar a atualização de segurança aos arquivos fonte da instalação do Windows.

    5.  Finalmente, você também pode clicar na guia **Versões Anteriores** e comparar as informações de arquivo da versão anterior do arquivo com as informações sobre a versão nova ou atualizada.
