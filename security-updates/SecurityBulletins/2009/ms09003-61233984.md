Microsoft Security Bulletin MS09-003 - Crítica
==============================================

Vulnerabilidades no Microsoft Exchange podem permitir a execução remota de código (959239)
------------------------------------------------------------------------------------------

Publicado: terça-feira, 10 de fevereiro de 2009 | Atualizado: terça-feira, 26 de maio de 2009

**Versão:** 3.0

### Informações Gerais

#### Sinopse

Esta atualização de segurança resolve duas vulnerabilidades relatadas em particular no Microsoft Exchange Server. A primeira vulnerabilidade pode permitir a execução remota de código se uma mensagem TNEF especialmente criada for enviada para o Microsoft Exchange Server. Um invasor que explorar essa vulnerabilidade com êxito pode assumir o controle total de um sistema afetado com privilégios de conta de serviço do Exchange Server. A segunda vulnerabilidade pode permitir negação de serviço se um comando MAPI especialmente criado for enviado para o Microsoft Exchange Server. Um invasor que explorar essa vulnerabilidade com êxito pode fazer com que o Serviço de Atendedor do Sistema do Microsoft Exchange, entre outros serviços que usam o provedor EMSMDB32, pare de responder.

Esta atualização de segurança é classificada como Crítica para todas as edições com suporte do Microsoft Exchange 2000 Server, Microsoft Exchange Server 2003, Microsoft Exchange Server 2007 e Cliente MAPI do Microsoft Exchange Server. Para obter mais informação, consulte a subseção **Software afetado e não afetado** nesta seção.

A atualização de segurança elimina as vulnerabilidades modificando a maneira como o Microsoft Exchange Server interpreta as mensagens TNEF e os comandos MAPI. Para obter mais informação sobre as vulnerabilidades, consulte a subseção Perguntas freqüentes relacionada à entrada da vulnerabilidade específica presente na próxima seção, **Informações sobre a vulnerabilidade**.

**Recomendação.** A Microsoft recomenda que os clientes apliquem a atualização imediatamente.

**Problemas conhecidos.** O [Artigo 959239 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/959239) documenta os problemas atualmente conhecidos que os clientes podem enfrentar ao instalar esta atualização de segurança e as soluções recomendadas. Quando problemas atualmente conhecidos e soluções recomendadas pertencem somente a lançamentos específicos deste software, este artigo fornece links para outros artigos.

#### Software afetado e não afetado

O software a seguir foi testado para determinar quais versões ou edições foram afetadas. O ciclo de vida do suporte das outras versões ou edições já terminou ou elas não são afetadas. Para determinar o ciclo de vida do suporte para sua versão ou edição de software, visite o site [Ciclo de vida do Suporte Microsoft.](http://go.microsoft.com/fwlink/?linkid=21742)

Perguntas freqüentes relacionadas a esta atualização de segurança  
-----------------------------------------------------------------
  
<span></span>
**Por que este boletim foi relançado em 26 de maio de 2009?**    
A Microsoft revisou este boletim de segurança para anunciar uma alteração de detecção para esta atualização de segurança. Em consequência da correção, a detecção agora oferece a atualização do Microsoft Exchange Server 2003 Service Pack 2 (KB959897) a sistemas executando o Exchange System Management Tools for Exchange Server 2003 nos quais nenhuma instância ativa do serviço do Exchange esteja sendo executada. Anteriormente, esta configuração não oferecia a atualização de segurança. Os clientes que já instalaram a atualização KB959897 com êxito em seus sistemas não necessitam reinstalá-la.
  
**Por que este boletim foi relançado em 16 de fevereiro de 2009?**    
A Microsoft relançou este boletim para adicionar o Cliente MAPI do Microsoft Exchange Server como software afetado. Nenhum outro pacote de atualização é afetado por este relançamento. Os clientes que estiverem executando todas as outras versões com suporte e afetadas do Microsoft Exchange Server que já tiverem aplicado os pacotes de atualização de segurança originais não precisam tomar nenhuma providência com relação a esses pacotes de atualização original.
  
A Microsoft também revisou este boletim para adicionar várias entradas à seção **Perguntas freqüentes relacionadas a esta atualização de segurança** relacionadas à atualização do Cliente MAPI do Microsoft Exchange Server e das ferramentas de Gerenciamento do Sistema do Exchange.
  
**Uso uma versão mais antiga do Cliente MAPI do Microsoft Exchange Server. Como atualizo para a versão 6.5.8069?**    
A Microsoft forneceu aos usuários do Cliente MAPI do Microsoft Exchange Server uma atualização gratuita para a versão 6.5.8069, disponível no [Centro de Download](http://www.microsoft.com/downloads/details.aspx?familyid=e17e7f31-079a-43a9-bff2-0a110307611e). Como resultado, todos os clientes que usam uma versão mais antiga do pacote do Cliente MAPI do Microsoft Exchange Server e do Collaboration Data Objects 1.2.1 devem atualizar para a versão 6.5.8069 para se protegerem das vulnerabilidades descritas neste boletim.
  
**O pacote de atualização do Cliente MAPI do Microsoft Exchange Server também inclui o CDO 1.2.1. O?CDO 1.2.1 é afetado pelas vulnerabilidades descritas neste boletim?**    
O download para o Cliente MAPI do Microsoft Exchange Server também inclui o pacote para o Collaboration Data Objects (CDO) 1.2.1. O?pacote do CDO 1.2.1 não é afetado pelas vulnerabilidades descritas neste boletim. O pacote é incluído no download para sua conveniência.
  
**Como sei se tenho o Cliente MAPI do Microsoft Exchange Server e o Collaboration Data Objects 1.2.1 instalados em meu sistema?**    
No Painel de Controle, procure na ferramenta **Adicionar ou Remover Programas** (Windows Server 2003 e anteriores) ou **Programas e Recursos** (Windows Vista e Windows Server 2008) a entrada "Messaging API e Collaboration Data Objects 1.2.1".
  
**Há dois downloads disponíveis no** **Centro deDownload** **para Cliente MAPI do Microsoft Exchange Server e Collaboration Data Objects 1.2.1 com o mesmo título. Há alguma diferença entre eles?**    
Não, não há nenhuma diferença. O Cliente MAPI pode ser baixado de qualquer uma das seguintes páginas:  
[Cliente MAPI do Microsoft Exchange Server e Collaboration Data Objects 1.2.1](http://www.microsoft.com/downloads/details.aspx?familyid=e17e7f31-079a-43a9-bff2-0a110307611e&displaylang=en)  
[Cliente MAPI do Microsoft Exchange Server e Collaboration Data Objects 1.2.1](http://www.microsoft.com/downloads/details.aspx?familyid=94274318-27c4-4d8d-9bc5-3e6484286b1f&displaylang=en)
  
**Tenho as ferramentas de Gerenciamento do Sistema do Exchange para Exchange Server 2003. Meu sistema está vulnerável?**    
As ferramentas de Gerenciamento do Sistema do Exchange para Exchange Server 2003 contêm o código vulnerável. No entanto, o código só pode ser explorado se o servidor também executar uma instância ativa do serviço do Exchange. Assim, os clientes que executam somente as ferramentas de Gerenciamento do Sistema do Exchange para Exchange Server 2003 e têm uma instância ativa do serviço do Exchange ativada devem instalar a atualização de segurança para Exchange Server 2003 Service Pack 2 (KB959897) para se protegerem contra as vulnerabilidades descritas neste boletim.
  
Se o serviço do Exchange for desativado, os vetores de ataque passíveis de exploração abordados neste boletim de segurança não ficarão expostos. No entanto, os clientes podem instalar a atualização de segurança para Exchange Server 2003 Service Pack 2 (KB959897) como uma medida de defesa profunda.
  
**Sou desenvolvedor de aplicativos terceirizados e recomendo aos clientes instalar as ferramentas de Gerenciamento do Sistema do Exchange para Exchange Server 2003 como um pré-requisito para usar meu aplicativo. Como eles podem atualizá-la?**    
Os clientes que executam somente as ferramentas de Gerenciamento do Sistema do Exchange para Exchange Server 2003 podem instalar a atualização de segurança para Exchange Server 2003 Service Pack 2 (KB959897) para se protegerem contra as vulnerabilidades descritas neste boletim.
  
**Instalei o Gerenciador do Sistema do Exchange para Windows.Vista. Meu sistema está vulnerável?**    
O [Gerenciador do Sistema do Exchange para Windows Vista](http://www.microsoft.com/downloads/details.aspx?familyid=3403d74e-8942-421b-8738-b3664559e46f&displaylang=en) é um download disponível para executar o Gerenciador do Sistema do Exchange para Exchange Server 2003 no sistema operacional Windows Vista. Embora, o Gerenciador do Sistema do Exchange para Windows Vista não seja afetado pelas vulnerabilidades descritas neste boletim, o Cliente MAPI do Microsoft Exchange Server e o Collaboration Data Objects 1.2.1 são um pré-requisito para instalar o Gerenciador do Sistema do Exchange para Windows Vista. Como resultado, os clientes que executam o Gerenciador do Sistema do Exchange para Windows Vista devem se assegurar de que o Cliente MAPI do Microsoft Exchange Server e o Collaboration Data Objects 1.2.1 em execução em seu sistema estejam atualizados para a versão mais recente.

**Qual é a diferença entre os modelos de serviço do Microsoft Exchange Server 2007 e do Microsoft Exchange Server 2003 e como a diferença causa impacto nas atualizações deste boletim de segurança?**    
Com o lançamento do Microsoft Exchange Server 2007, o Microsoft Exchange passou para um novo modelo de serviço baseado em comentários do cliente e uniformidade com outros modelos de serviço de produtos da Microsoft. As atualizações do Exchange Server 2007 são cumulativas no nível de atualização oferecido e no nível de arquivo individual, enquanto as atualizações do Exchange Server 2003 são cumulativas somente no nível de arquivo.
  
Para obter uma explicação mais detalhada do modelo de serviço do Microsoft Exchange, consulte a [documentação do Microsoft Exchange Server 2007](http://go.microsoft.com/fwlink/?linkid=89577). Se tiver alguma dúvida sobre o novo modelo de serviço do Exchange, contate os [Serviços de Suporte ao Produto Microsoft.](http://support.microsoft.com/common/international.aspx)
  
**Preciso instalar o pacote de conjunto de atualizações paraservidores baseados no Exchange Server 2007 em uma determinada sequência?**    
Nossa infra-estrutura de teste ajuda a garantir que nossas atualizações funcionem em várias funções de servidor. Portanto, você não precisa aplicar um conjunto de atualizações em uma sequência requerida para servidores baseados no Exchange que estejam executando funções diferentes. No entanto, você deve aplicar um conjunto de atualizações a cada servidor baseado no Exchange Server 2007 em seu ambiente. Isto acontece porque os conjuntos de atualizações não são divididos para uso com diferentes funções do Exchange ou para uso com determinadas configurações de arquivo.
  
Se você for um cliente do CAS Proxy Deployment Guidance e se tiver implantado proxy entre CAS, aplique o conjunto de atualizações aos servidores de acesso para clientes conectados à Internet antes de aplicá-lo aos servidores de acesso para clientes não conectados à Internet. Para outras configurações do Exchange Server 2007, a ordem na qual você aplica o conjunto de atualizações aos servidores não é importante.
  
Para obter mais informações sobre proxy entre CAS, consulte [Entendendo o que é proxy e redirecionamento](http://technet.microsoft.com/en-us/library/bb310763.aspx).

Esta atualização contém suporte para diversas vulnerabilidades porque as modificações necessárias para corrigir esses problemas foram feitas em arquivos relacionados. Em vez de instalar várias atualizações quase iguais, os clientes poderão instalar apenas esta atualização

Vulnerabilidade de corrupção de memória - CVE-2009-0098  
-------------------------------------------------------
  
<span></span>
Uma vulnerabilidade de execução remota de código existe na maneira como o Microsoft Exchange Server decodifica os dados em formato TNEF (Transport Neutral Encapsulation Format) de uma mensagem.
  
Para exibir essa vulnerabilidade como uma entrada padrão na lista Common Vulnerabilities and Exposures, consulte [CVE-2009-0098](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0098).
  
#### Fatores atenuantes para a vulnerabilidade de corrupção de Memória - CVE-2009-0098
  
A Microsoft não identificou nenhum fator atenuante para essa vulnerabilidade.
  
#### Soluções alternativas para a vulnerabilidade de corrupção de Memória - CVE-2009-0098
  
A solução alternativa refere-se a uma configuração ou alteração de configuração que não corrige a vulnerabilidade subjacente mas que ajudaria a bloquear vetores de ataque conhecidos antes de aplicar a atualização. A Microsoft testou as soluções alternativas e estados a seguir como parte do cenário que implica se uma solução alternativa reduziria funcionalidade:
  
-   **Bloquear o MS-TNEF no Microsoft Exchange Server para ajudar a proteger contra tentativas de exploração dessa vulnerabilidade através de emails SMTP**
  
    O bloqueio do tipo de conteúdo MIME do aplicativo/ms-tnef pode ajudar a proteger os servidores Exchange e outros programas afetados contra tentativas de exploração dessa vulnerabilidade se clientes não conseguirem instalar a atualização de segurança disponível.
  
    Os sistemas podem ser configurados para bloquear o recebimento de determinados tipos de arquivos como anexos de email. As mensagens de email codificadas pelo TNEF da Microsoft, normalmente conhecidas como mensagens RTF (Rich Text Format) podem conter objetos OLE mal-intencionados. Essas mensagens de email contêm um anexo de arquivo que armazena a informação do TNEF. Esse anexo de arquivo é normalmente denominado Winmail.dat. Bloquear esse arquivo e bloquear o tipo MIME do ms-tnef pode ajudar a proteger servidores Exchange e outros programas afetados contra tentativas de explorar esta vulnerabilidade, caso os clientes não instalem a atualização de segurança disponível. Para ajudar a proteger um computador Exchange Server contra ataques por SMTP, bloqueie o arquivo Winmail.dat e todo o conteúdo de tipo MIME de aplicativo/ms-tnef antes que ele chegue ao computador Exchange Server.
  
    **Observação** Você não pode atenuar essa vulnerabilidade configurando a opção de **formato Rich Text do Exchange** no Exchange Server como **Nunca usado** nem desabilitando o processamento de TNEF no Registro.
  
    **Observação** O Exchange oferece suporte a outros protocolos de serviço de mensagens, como o X.400, que não são protegidos por essas soluções alternativas. Recomendamos que os administradores exijam autenticação em todos os outros protocolos de transporte de mensagens e clientes para ajudar a impedir ataques usando esses protocolos.
  
    **Observação** Filtrar somente anexos que tenham o nome de arquivo Winmail.dat pode não ser suficiente para ajudar a proteger seu sistema. Um anexo de arquivo mal-intencionado pode ter um nome de arquivo diferente que poderia ser processado pelo computador com o Exchange Server. Para ajudar a proteger contra mensagens de email mal-intencionadas, bloqueie todo o conteúdo de tipo MIME de aplicativo/ms-tnef.
  
    Há muitas formas de bloquear o arquivo Winmail.dat e outro conteúdo de TNEF. Eis algumas sugestões:
  
    -   Você pode usar o screener de mensagens SMTP do ISA Server 2000 para bloquear todos os anexos de arquivo ou para bloquear somente o arquivo Winmail.dat. O bloqueio de todos os anexos de arquivo oferece a melhor proteção para esse problema caso você use o ISA Server 2000, pois esse programa não oferece suporte a bloqueio de conteúdo com base em tipos de conteúdo MIME. Para obter mais informações, consulte o [artigo 315132 do Microsoft Knowledge Base](http://support.microsoft.com/?id=315132).  
    -   Você pode usar o Filtro de SMTP do ISA Server 2000 para bloquear todos os anexos de arquivo ou para bloquear somente o arquivo Winmail.dat. O bloqueio de todos os anexos de arquivo oferece a melhor proteção para esse problema caso você use o ISA Server 2000, pois esse programa não oferece suporte a bloqueio de conteúdo com base em tipos de conteúdo MIME. Para obter mais informações, consulte o [artigo 320703 do Microsoft Knowledge Base](http://support.microsoft.com/?id=320703).  
    -   Você pode usar o filtro e o screener de mensagens SMTP do ISA Server 2004 para bloquear todos os anexos de arquivo ou somente o arquivo Winmail.dat. O bloqueio de todos os anexos de arquivo oferece a melhor proteção contra esse problema se você usar o ISA Server 2004, pois o programa não oferece suporte a bloqueio de conteúdo com base em tipos de conteúdo MIME. Para obter mais informações, consulte o [artigo 888709 do Microsoft Knowledge Base](http://support.microsoft.com/?id=888709).  
    -   Você pode usar filtragem de anexo no Microsoft Exchange Server 2007 para aplicar filtros no nível de servidor, a fim de controlar os anexos recebidos pelos usuários. Consulte [Filtragem de anexo](http://technet.microsoft.com/en-us/library/bb124399.aspx) para obter mais informações.  
    -   Você pode usar filtros de email de terceiros para bloquear todo o conteúdo/aplicativos do tipo MIME do ms-tnef antes que sejam enviados ao computador com o Exchange Server ou a um aplicativo vulnerável.
  
    **Impacto da solução alternativa:** Se os anexos TNEF forem bloqueados, as mensagens de email formatadas como RTF não serão recebidas corretamente. Em alguns casos, os usuários podem receber mensagens de email em branco em vez da mensagem original formatada como RTF. Em outros casos, os usuários podem não receber nenhuma mensagem de email formatada como RTF. O bloqueio de anexos TNEF não afetará as mensagens de email formatadas como HTML ou como texto sem formatação.
  
#### Perguntas frequentes sobre a vulnerabilidade de corrupção de memória - CVE-2009-0098
  
**Qual é o escopo da vulnerabilidade?**    
Essa é uma vulnerabilidade de execução remota de código. Um invasor que explorar com êxito essa vulnerabilidade poderá assumir o controle total do sistema afetado com privilégios de conta de serviço do Exchange Server, quando um usuário abrir ou exibir uma mensagem de email especialmente criada e enviada no formato TNEF ou quando o serviço Armazenamento de Informações do Microsoft Exchange processar a mensagem especialmente criada. Um invasor poderá instalar programas, visualizar, alterar ou excluir dados, ou ainda criar novas contas.
  
**O que provoca a vulnerabilidade?**    
O Exchange Server não decodifica corretamente uma mensagem recebida no formato TNEF.
  
**O que é TNEF?**    
TNEF (Transport Neutral Encapsulation) é um formato utilizado pelo Microsoft Exchange Server durante o envio de mensagens formatadas como RTF (Rich Text Format). Quando o Microsoft Exchange envia uma mensagem para outro cliente de email da Microsoft, ele extrai todas as informações de formatação e as codifica em um bloco TNEF especial. Em seguida, envia a mensagem em duas partes: a mensagem de texto com o formatação removida e as instruções de formatação no bloco TNEF. No lado de recepção, um cliente de email da Microsoft processa o bloco TNEF e reformata a mensagem
  
**Para que um invasor pode usar a vulnerabilidade?**    
Um invasor que explorar essa vulnerabilidade com êxito pode assumir o controle total de um sistema afetado com privilégios de conta de serviço do Exchange Server. Um invasor poderá instalar programas, visualizar, alterar ou excluir dados, ou ainda criar novas contas.
  
**De que forma o invasor pode explorar a vulnerabilidade?**    
Um invasor pode explorar a vulnerabilidade enviando uma mensagem TNEF especialmente criada para um Microsoft Exchange Server.
  
**Quais são os sistemas que mais correm riscos com a vulnerabilidade?**    
Os servidores Microsoft Exchange correm risco.
  
**O que a atualização faz?**    
A atualização corrige a maneira como esse Microsoft Exchange Server interpreta as propriedades TNEF específicas.
  
**Quando esse boletim de segurança foi lançado, essa vulnerabilidade já tinha sido divulgada publicamente?**    
Não. A Microsoft recebeu informações sobre essa vulnerabilidade por meio de divulgação responsável.
  
**Quando este boletim de segurança foi lançado, a Microsoft tinha recebido algum relatório informando que essa vulnerabilidade estava sendo explorada?**    
Não. A Microsoft não recebeu nenhuma informação indicando que essa vulnerabilidade tenha sido usada publicamente para atacar clientes e não testemunhou nenhum exemplo de publicação de código de verificação de conceito, quando este boletim de segurança foi lançado pela primeira vez.
  
Vulnerabilidade de processamento literal - CVE-2009-0099  
--------------------------------------------------------
  
<span></span>
Existe uma vulnerabilidade de negação de serviço no provedor EMSMDB2 (Electronic Messaging System Microsoft Data Base, versão de 32 bits) devido à forma como ele manipula comandos MAPI inválidos. Um invasor pode explorar a vulnerabilidade enviando um comando MAPI especialmente criado ao aplicativo usando o provedor EMSMDB32. Um invasor que explorar com êxito essa vulnerabilidade poderá fazer com que o aplicativo pare de responder.
  
A vulnerabilidade de negação de serviço também afeta o Atendedor do Sistema do Microsoft Exchange, já que ele usa o provedor EMSMDB32. O Atendedor do Sistema do Microsoft Exchange é um dos serviços principais no Microsoft Exchange e executa uma variedade de funções relacionadas à manutenção contínua do sistema Exchange.
  
Para exibir essa vulnerabilidade como uma entrada padrão na lista Common Vulnerabilities and Exposures, consulte [CVE-2009-0099](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0099).

A Microsoft não identificou nenhum fator atenuante para essa vulnerabilidade.

A Microsoft não identificou nenhuma solução alternativa para esta vulnerabilidade.
  
#### Perguntas frequentes sobre a vulnerabilidade de processamento literal - CVE-2009-0099
  
**Qual é o escopo da vulnerabilidade?**    
Essa é uma vulnerabilidade de negação de serviço (site em inglês). Um invasor que explorar essa vulnerabilidade com êxito pode fazer com que o Serviço de Atendedor do Sistema do Microsoft Exchange, entre outros serviços que usam o provedor EMSMDB32, pare de responder.
  
**O que provoca a vulnerabilidade?**    
A vulnerabilidade é causada pela manipulação incorreta de um comando no provedor EMSMDB32 pelo Microsoft Exchange.
  
**O que é MAPI?**  
MAPI é um conjunto de funções que os aplicativos habilitados para email e compatíveis com email usam para criar, manipular, transferir e armazenar mensagens de email. Ele oferece aos desenvolvedores de aplicativos as ferramentas para definirem a finalidade e o conteúdo das mensagens de email, bem como a flexibilidade de gerenciamento de mensagens de email armazenadas. MAPI também oferece uma interface comum que os desenvolvedores de aplicativos podem usar para criar aplicativos habilitados para email e compatíveis com email independentemente do sistema de mensagens subjacente. Consulte o artigo sobre [Referência da biblioteca MFC para MAPI](http://msdn.microsoft.com/en-us/library/245421h5(vs.80).aspx) (pode estar em inglês) para obter mais informações.
  
**O que é EMSMDB32?**  
O EMSMDB32 (Electronic Messaging System Microsoft Data Base, versão de 32 bits) refere-se ao provedor de Transporte do Exchange que implementa um provedor de transporte e um de armazenamento de mensagens para o MAPI. Ele oferece a capacidade de envio de mensagens para o Exchange Server e de leitura (e possivelmente de gravação) em um armazenamento do Exchange. Consulte [Como o Outlook, CDO, MAPI e provedores funcionam juntos](http://technet.microsoft.com/en-us/library/aa996249.aspx) (pode estar em inglês) para obter mais informações sobre como o MAPI e provedores funcionam juntos.
  
**Para que um invasor pode usar a vulnerabilidade?**    
Um invasor que explorar com êxito essa vulnerabilidade poderá fazer com que o aplicativo ou serviço que estiver usando o provedor EMSMDB32 pare de responder. Um dos serviços que pode parar de responder é o serviço Atendedor do Sistema de Microsoft Exchange, porque ele usa o provedor EMSMDB32. O Atendedor do Sistema Microsoft Exchange executa uma variedade de funções relacionadas à manutenção e processamento contínuos do sistema Exchange, tais como a geração de listas de endereços, catálogos de endereços offline e recursos de consulta em diretórios. Essas tarefas serão afetadas se o serviço Atendedor do Sistema Exchange parar de funcionar.
  
**De que forma o invasor pode explorar a vulnerabilidade?**    
Um invasor pode tentar explorar a vulnerabilidade enviando um comando MAPI especialmente criado para um Microsoft Exchange Server. O invasor que explorar com êxito essa vulnerabilidade poderá fazer com que o serviço de email pare de responder.
  
**Quais são os sistemas que mais correm riscos com a vulnerabilidade?**    
Os sistemas de servidor Exchange que utilizam o provedor EMSMDB correm mais risco com essa vulnerabilidade.
  
**O que a atualização faz?**    
A atualização elimina a vulnerabilidade modificando o modo como o Microsoft Exchange Server manipula comandos MAPI malformados.
  
**Quando esse boletim de segurança foi lançado, essa vulnerabilidade já tinha sido divulgada publicamente?**    
Não. A Microsoft recebeu informações sobre essa vulnerabilidade por meio de divulgação responsável.
  
**Quando este boletim de segurança foi lançado, a Microsoft tinha recebido algum relatório informando que essa vulnerabilidade estava sendo explorada?**    
Não. A Microsoft não recebeu nenhuma informação indicando que essa vulnerabilidade tenha sido usada publicamente para atacar clientes e não testemunhou nenhum exemplo de publicação de código de verificação de conceito, quando este boletim de segurança foi lançado pela primeira vez.

**Verificando a aplicação da atualização**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma atualização de segurança foi aplicada em um sistema afetado, é possível utilizar a ferramenta MBSA (Microsoft Baseline Security Analyzer). Consulte a seção **Orientação e ferramentas de detecção e implantação** neste boletim para obter mais informações.

-   **Verificação de Versão do Arquivo**

    Como existem várias versões do Microsoft Windows, as etapas a seguir podem ser diferentes no seu sistema. Se isso acontecer, consulte a documentação do produto para completar estas etapas.

    1.  Clique em **Iniciar** e insira um nome de arquivo de atualização em **Iniciar Pesquisa**.
    2.  Quando o arquivo for exibido em **Programas**, clique com o botão direito do mouse no nome do arquivo e clique em **Propriedades**.
    3.  Na guia **Geral**, compare o tamanho do arquivo com as tabelas de informações sobre arquivos fornecidas no artigo da KB do boletim.
    4.  Você também pode clicar na guia **Detalhes** e comparar as informações, como versão do arquivo e data da última modificação, com as tabelas de informações de arquivos fornecidas no artigo da KB do boletim.
    5.  Finalmente, você também pode clicar na guia **Versões Anteriores** e comparar as informações de arquivo da versão anterior do arquivo com as informações sobre a versão nova ou atualizada.

-   **Verificação da Chave do Registro**

    Você também pode ser capaz de verificar os arquivos que esta atualização de segurança instalou examinando as chaves do Registro na **Tabela de referência** nesta seção.

    Essas chaves do Registro podem não conter uma lista completa dos arquivos instalados. Além disso, talvez essas chaves do Registro não sejam criadas corretamente se um administrador ou um OEM integrar ou adicionar a atualização de segurança aos arquivos fonte da instalação do Windows.
