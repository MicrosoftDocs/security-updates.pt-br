Microsoft Security Bulletin MS10-041 - Importante
=================================================

Vulnerabilidade no Microsoft .NET Framework pode permitir falsificação (981343)
-------------------------------------------------------------------------------

Publicado: terça-feira, 8 de junho de 2010 | Atualizado: quarta-feira, 30 de junho de 2010

**Versão:** 1.2

### Informações Gerais

#### Sinopse

Esta atualização de segurança elimina uma vulnerabilidade divulgada publicamente no Microsoft .NET Framework. A vulnerabilidade pode permitir a falsificação de dados de conteúdo XML assinado, sem ser detectada. Em aplicativos personalizados, o impacto de segurança depende de como o conteúdo assinado é usado no aplicativo específico. Os cenários em que as mensagens XML assinadas são transmitidas através de um canal seguro (como SSL) não são afetados por esta vulnerabilidade.

Esta atualização de segurança foi classificada como Importante para todas as versões afetadas do Microsoft .NET Framework para Microsoft Windows 2000, Windows XP, Windows Server 2003, Windows Vista, Windows Server 2008, Windows 7 e Windows Server 2008 R2. Para obter mais informações, consulte a subseção **Softwares afetados e não afetados**, nesta seção.

A atualização de segurança elimina a vulnerabilidade mudando a maneira como a recomendação XMLDsig foi implementada no Microsoft .NET Framework. Para obter mais informações sobre a vulnerabilidade**,** consulte a subseção Perguntas frequentes relacionada à entrada da vulnerabilidade específica, localizada na próxima seção, **Informações sobre a vulnerabilidade**.

**Recomendação.**  A maioria dos clientes tem o recurso de atualizações automáticas habilitado e não precisará tomar nenhuma providência porque esta atualização de segurança será baixada e instalada automaticamente. Os clientes que não têm o recurso de atualizações automáticas habilitado precisarão verificar as atualizações e instalar esta atualização manualmente. Para obter informações sobre opções de configuração específicas em atualizações automáticas, consulte o [Artigo 294871 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/294871).

Para instalações feitas por administradores ou empresas, ou por usuários finais que desejam instalar esta atualização de segurança manualmente, a Microsoft recomenda que os clientes apliquem a atualização assim que possível usando software de gerenciamento de atualização ou verificando se há atualizações no serviço [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Consulte também a seção **Orientação e ferramentas de detecção e implantação** mais adiante neste boletim.

**Problemas conhecidos.** O [Artigo 981343 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/981343) documenta os problemas atualmente conhecidos que os clientes podem enfrentar ao instalar esta atualização de segurança. O artigo também documenta as soluções recomendadas para esses problemas.

#### Softwares afetados e não afetados

Os softwares a seguir foram testados para determinar quais versões ou edições foram afetadas. O ciclo de vida do suporte das outras versões ou edições já terminou ou elas não são afetadas. Para determinar o ciclo de vida do suporte para sua versão ou edição de software, visite o site [Ciclo de vida do Suporte Microsoft.](http://go.microsoft.com/fwlink/?linkid=21742)

Perguntas frequentes relacionadas a esta atualização de segurança
-----------------------------------------------------------------

**Por que este boletim foi revisado em 22 de junho de 2010?**  
A Microsoft revisou este boletim para anunciar uma alteração da lógica de detecção para corrigir um problema de listagem incorreta da atualização de segurança de 8 de junho de 2010 para o Microsoft .NET Framework 3.5 (KB982865) como "instalada" pelo Microsoft Baseline Security Analyzer (MBSA) em sistemas com o Microsoft .NET Framework 3.5 Service Pack 1 instalado. No entanto, a aplicabilidade da atualização de segurança para o Microsoft .NET Framework 2.0 Service Pack 2 e Microsoft .NET Framework 3.5 Service Pack 1 (KB979909) está sendo detectada corretamente pelo MBSA. Esta é apenas uma alteração de detecção que não afeta os arquivos contidos na atualização inicial. Os clientes que atualizaram seus sistemas com êxito não necessitam reinstalar esta atualização.

Esta alteração na detecção pode fazer com que a atualização KB982865 seja apresentada como uma nova atualização para clientes que estejam usando o Windows Server Update Services (WSUS). Se você não tiver regras de aprovação automática definidas, talvez precise aprovar a atualização manualmente mais uma vez. Não voltar a aprovar a atualização pode fazer com que a atualização continue a ser oferecida aos sistemas, mesmo que a atualização não seja aplicável. Para obter mais informações sobre esta alteração na detecção, consulte o [Artigo 894199 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/894199).

**O que é uma vulnerabilidade de falsificação de dados?**  
Em segurança da informação, uma vulnerabilidade de falsificação de dados pode permitir a modificação mal-intencionada de dados. Exemplos incluem alterações não autorizadas feitas a dados persistentes, como os dados mantidos em um banco de dados, e a alteração de dados enquanto fluem entre dois computadores em uma rede aberta, como a Internet. Especificamente para esta vulnerabilidade, se uma assinatura criptográfica eficiente for usada para assinar digitalmente os dados, esses dados forem modificados depois de assinados, e a verificação da assinatura digital falhar, isso indica falsificação. Se a verificação da assinatura for bem-sucedida apesar do fato de os dados terem sido falsificados, isto também indica falsificação.

**Como determino que versão do Microsoft .NET Framework está instalada?**  
Você pode instalar e executar várias versões do .Net Framework em um sistema, além de poder instalar as versões em qualquer ordem. Há várias maneiras de determinar quais versões do .Net Framework estão instaladas no momento. Para obter mais informações, consulte o [Artigo 318785 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/318785) ou o artigo [Determining Which Version of the .NET Framework Is Installed](http://msdn.microsoft.com/en-us/library/y549e41e.aspx) (em inglês) do MSDN.

**Por que o Microsoft .NET Framework 3.5 e o Microsoft .NET Framework 3.5 Service Pack 1 são afetados em alguns sistemas operacionais com suporte?**  
O Microsoft .NET Framework 3.5 inclui os subcomponentes Microsoft .NET Framework 2.0 Service Pack 1 e Microsoft .NET Framework 3.0 Service Pack 1. O Microsoft .NET Framework 3.5 Service Pack 1 contém os subcomponentes Microsoft .NET Framework 2.0 Service Pack 2 e Microsoft .NET Framework 3.0 Service Pack 2. Em alguns sistemas operacionais, o Microsoft .NET Framework 3.5 e o Microsoft .NET Framework 3.5 Service Pack 1 podem instalar esses subcomponentes vulneráveis.

**Tenho uma versão do Microsoft .NET Framework, que não está listada neste boletim, instalada no meu sistema. Minha configuração é afetada por esta vulnerabilidade?**  
Os softwares afetados listados neste boletim foram testados para determinar que versões são afetadas. Outras versões passaram do seu ciclo de vida de suporte ou não existe suporte para elas. Os clientes que têm uma versão sem suporte do Microsoft .NET Framework instalada em seu sistema são aconselhados a desinstalar essa versão do Microsoft .NET Framework e atualizar para uma versão mais recente do Microsoft .NET Framework.

O Microsoft .NET Framework pode ser desinstalado com a ferramenta **Adicionar ou Remover Programas**, localizada no Painel de Controle. Para obter mais informações sobre a remoção de versões específicas do Microsoft .NET Framework, consulte o [Artigo 320122 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/320112), o [Artigo 824643 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/824643) e o [Artigo 908077 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/908077).

**Esta atualização contém alterações relacionadas à segurança da funcionalidade?**  
Sim. Além das alterações listadas na seção **Informações sobre a vulnerabilidade** deste boletim, esta atualização inclui uma alteração de defesa profunda ao recurso de validação de solicitação ASP.NET para eliminar um problema que pode permitir a um invasor ignorar uma medida básica de defesa profunda que fica ativada por padrão em sites habilitados para ASP.NET.

O problema é causado porque o recurso de validação de solicitação no ASP.NET não verifica adequadamente uma sequência específica de caracteres. O recurso de validação de solicitação ASP.NET não pode substituir uma camada de validação efetiva na restrição de variáveis de entrada não confiáveis. Os desenvolvedores que desejarem saber mais sobre os recursos de segurança fornecidos pelo ASP.NET a aplicativos Web podem consultar o artigo [Take Advantage of ASP.NET Built-in Features to Fend Off Web Attacks](http://msdn2.microsoft.com/en-us/library/ms972969.aspx) (em inglês) do MSDN.

Se um site não tem uma camada de validação eficiente para restringir entradas de usuário não confiáveis, um invasor que explore este problema com êxito pode injetar conteúdo arbitrário, inclusive conteúdo ASP.NET no site afetado. Esta alteração de defesa profunda atenua o problema que foi relatado em particular.

**O que é a validação de solicitação ASP.NET?**  
Esta alteração de defesa profunda ao recurso de validação de solicitação ASP.NET executa a validação básica da entrada em sites que executam o ASP.NET. No entanto, o recurso de validação de solicitação ASP.NET não pode substituir uma camada de validação eficaz que restrinja variáveis de entrada não confiáveis. Os desenvolvedores que desejarem saber mais sobre os recursos de segurança fornecidos pelo ASP.NET a aplicativos Web podem consultar os artigos [Take Advantage of ASP.NET Built-in Features to Fend Off Web Attacks](http://msdn2.microsoft.com/en-us/library/ms972969.aspx) e [Do Not Rely on ASP.NET Request Validation](http://msdn.microsoft.com/en-us/library/ms998264.aspx) (em inglês) do MSDN.

**Como sei se meu aplicativo Web ASP.NET é afetado por esta alteração de defesa profunda?**  
Aplicativos Web desenvolvidos em ASP.NET que restringem todas as variáveis de entrada não confiáveis a uma gama de valores ou caracteres esperados não são afetados. Para obter mais informações sobre a proteção dos aplicativos Web desenvolvidos em ASP.NET, consulte o [Artigo 815155 (em inglês) da Microsoft Knowledge Base.](http://support.microsoft.com/kb/815155)

**O que é ASP.NET?**  
O ASP.NET é uma coleção de tecnologias no Microsoft .NET Framework que permite aos desenvolvedores criar aplicativos da Web e serviços da Web em XML.

Diferentemente das páginas da Web tradicionais, que usam uma combinação de HTML estático e scripts, o ASP.NET usa páginas compiladas determinadas por eventos. Por ser um ambiente de aplicativos baseado na Web, exigindo um servidor Web subjacente para fornecer funcionalidade HTTP básica, o ASP.NET é executado sobre o Internet Information Services (IIS). Para obter mais informações, consulte o [Site Oficial do Microsoft ASP.NET](http://www.asp.net/).

**Por que este problema é abordado como uma medida de defesa profunda?**  
A validação de solicitação ASP.NET está sendo abordada como uma alteração de defesa profunda, que pode ser usada como uma medida de prevenção extra além da validação de entrada do próprio desenvolvedor. Somente o desenvolvedor pode definir o que constitui uma entrada boa para um aplicativo específico. Os recursos de defesa profunda não são projetados para serem usados como única medida, ainda que várias camadas do tipo possam ajudar consideravelmente a evitar que um invasor comprometa a segurança do sistema em questão. Portanto, se a validação de solicitação ASP.NET ignorar uma sequência específica de caracteres, o impacto máximo de segurança deste problema não pode ser maior do que se este recurso estivesse completamente desabilitado.

**O que é defesa profunda?**  
Em segurança da informação, defesa profunda se refere a uma abordagem em que várias camadas de defesa são estabelecidas para ajudar a impedir que invasores comprometam a segurança de uma rede ou sistema.

**Uso uma versão mais antiga do software discutido neste boletim de segurança. O que devo fazer?**  
Os softwares afetados listados neste boletim foram testados para determinar que versões são afetadas. Outras versões passaram seu ciclo de vida de suporte. Os clientes que possuem versões anteriores do software devem priorizar a migração para as versões com suporte, a fim de evitar uma possível exposição a vulnerabilidades. Para determinar o ciclo de vida do suporte para sua versão de software, consulte [Selecione um Produto para Obter Informações do Ciclo de Vida](http://go.microsoft.com/fwlink/?linkid=169555). Para obter mais informações sobre service packs para esses lançamentos de software, consulte [Lifecycle Supported Service Packs](http://go.microsoft.com/fwlink/?linkid=89213).

Os clientes que precisarem de suporte adicional para software mais antigo deverão entrar em contato com o representante da equipe de contas da Microsoft, o gerente técnico da conta ou o representante do parceiro Microsoft apropriado para obter opções de suporte personalizadas. Os clientes sem um contrato Alliance, Premier ou Authorized podem entrar em contato com o escritório de vendas local da Microsoft. Para obter informações de contato, visite o site de [Informações internacionais da Microsoft](http://go.microsoft.com/fwlink/?linkid=33329), selecione o país na lista de informações de contato e, em seguida, clique em **Go** para ver uma lista de telefones. Ao ligar, peça para falar com o gerente de vendas local de Suporte Premier. Vulnerabilidade de anulação de autenticação de truncamento HMAC na assinatura XML - CVE-2009-0217
-------------------------------------------------------------------------------------------------

Existe uma vulnerabilidade de falsificação de dados no Microsoft .NET Framework que pode permitir a um invasor falsificar conteúdo XML assinado sem ser detectado. Em aplicativos personalizados, o impacto de segurança depende do cenário de uso específico. Os cenários em que as mensagens XML assinadas são transmitidas através de um canal seguro (como SSL) não são afetados por esta vulnerabilidade.

Para exibir esta vulnerabilidade como uma entrada padrão na lista Common Vulnerabilities and Exposures, consulte [CVE-2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217).

#### Fatores atenuantes da vulnerabilidade de anulação de autenticação de truncamento HMAC na assinatura XML - CVE-2009-0217

A atenuação refere-se a uma configuração, configuração comum ou prática recomendada geral, existindo num estado padrão, que possa reduzir a gravidade de exploração de uma vulnerabilidade. Os fatores de atenuação a seguir podem ser úteis em sua situação:

-   O Microsoft Office implementa a verificação de assinatura XML de uma maneira que não pode ser explorada por um invasor que tente explorar esta vulnerabilidade de falsificação de dados.

#### Soluções alternativas para a vulnerabilidade de anulação de autenticação de truncamento HMAC na assinatura XML - CVE-2009-0217

A solução alternativa refere-se a uma configuração ou alteração de configuração que não corrige a vulnerabilidade subjacente mas que ajudaria a bloquear vetores de ataque conhecidos antes de aplicar a atualização. A Microsoft testou as soluções alternativas e estados a seguir como parte do cenário que implica se uma solução alternativa reduziria funcionalidade:

-   **Use um canal seguro para transmitir informações confidenciais**

    Se seu aplicativo transmite dados XML confidenciais ou dados XML críticos à segurança de seu aplicativo, use um canal seguro para transmitir essas informações. Por exemplo, se você usa um Servidor de informações da Internet, exija o SSL para aplicativos confidenciais ou críticos à segurança.

#### Perguntas frequentes sobre a vulnerabilidade de anulação de autenticação de truncamento HMAC na assinatura XML - CVE-2009-0217

**Qual é o escopo da vulnerabilidade?**  
Esta vulnerabilidade é uma vulnerabilidade de falsificação de dados que pode ser usada para ignorar uma assinatura criptográfica.

**O que provoca a vulnerabilidade?**  
A vulnerabilidade é causada pela maneira como a recomendação XMLDsig (XML Signature Syntax and Processing) do W3C foi implementada no Microsoft .NET Framework. Para obter mais informações sobre XMLDsig, consulte a Recomendação W3C, [XML Signature Syntax and Processing (Second Edition)](http://www.w3.org/tr/xmldsig-core/) ou consulte a seção "E03 HMAC truncation (CVE-2009-0217)" na [Errata for XML Signature Second Edition](http://www.w3.org/2008/06/xmldsigcore-errata.html).

**O que é o Microsoft .NET Framework?**  
O Microsoft .NET Framework é um componente do sistema operacional Microsoft Windows que possibilita a criação e execução de aplicativos de software e serviços Web. Inclui tecnologias para serviços Web e aplicativos Web (ASP.NET), acesso a dados (ADO.NET), aplicativos de smart client (Windows Forms) e muitos outros.

**O que é HMAC?**  
Um HMAC (Hash-based Message Authentication Code) pode ser usado para determinar se uma mensagem enviada por um canal não seguro foi falsificada, por meio da verificação da integridade e autenticidade da mensagem, desde que o remetente e o destinatário compartilhem uma chave secreta.

**O que é XML?**  
Extensible Markup Language, ou XML, é uma linguagem de marcação que fornece um formato para descrever dados estruturados. Extensible Markup Language (XML) é uma especificação do W3C (World Wide Web Consortium) e um subconjunto da SGML (Standard Generalized Markup Language). Para obter mais informações sobre XML, consulte a [home page de XML](http://msdn.microsoft.com/xml/) da página do MSDN Data Developer Center ou a Recomendação do W3C, [Extensible Markup Language (XML) 1.0 (Fifth Edition)](http://www.w3.org/tr/rec-xml/).

**Para que um invasor pode usar a vulnerabilidade?**  
Um invasor que explore com êxito esta vulnerabilidade pode ignorar certas assinaturas criptográficas e, como resultado, falsificar conteúdo XML assinado sem que o destinatário detecte as alterações. Se a mensagem for totalmente alterada com relação ao seu significado original, isso também pode constituir falsificação. Em situações em que um desenvolvedor programou um aplicativo que conta com conteúdo XML assinado por HMAC, o impacto de segurança depende do cenário de uso específico a esse aplicativo.

**De que forma o invasor pode explorar a vulnerabilidade?**  
Um invasor precisa enviar conteúdo XML especialmente criado a um sistema vulnerável. O Microsoft Office implementa a verificação de assinatura XML de uma maneira que não pode ser explorada por um invasor.

**Quais são os sistemas que mais correm riscos com a vulnerabilidade?**  
Os sistemas ou aplicativos que contam com o HMAC (Hash-based Message Authentication Code) para operações criptográficas são os que correm mais risco. O Microsoft Office implementa a verificação de assinatura XML de uma maneira que não pode ser explorada por um invasor que tente explorar a vulnerabilidade.

**O que a atualização faz?**  
A atualização aborda a maneira como a recomendação XMLDsig foi implementada no Microsoft .NET Framework.

**Quando esse boletim de segurança foi lançado, essa vulnerabilidade já tinha sido divulgada publicamente?**  
Sim. Essa vulnerabilidade foi divulgada publicamente. Ela recebeu o número [CVE-2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217) da lista Common Vulnerability and Exposure.

**Quando este boletim de segurança foi lançado, a Microsoft tinha recebido algum relatório informando que essa vulnerabilidade estava sendo explorada?**  
Não. A Microsoft não recebeu nenhuma informação indicando que essa vulnerabilidade tenha sido usada publicamente para atacar clientes quando este boletim de segurança foi lançado pela primeira vez.

**Verificando a aplicação da atualização**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma atualização de segurança foi aplicada em um sistema afetado, é possível utilizar a ferramenta MBSA (Microsoft Baseline Security Analyzer). Consulte a seção **Orientação e ferramentas de detecção e implantação** neste boletim para obter mais informações.

-   **Verificação de Versão do Arquivo**

    Como existem várias versões do Microsoft Windows, as etapas a seguir podem ser diferentes no seu sistema. Se isso acontecer, consulte a documentação do produto para completar estas etapas.

    1.  Clique em **Iniciar** e, em seguida, clique em **Pesquisar**.
    2.  No painel **Resultados da Pesquisa**, clique em **Todos os arquivos e pastas**, em **Search Companion**.
    3.  Na caixa **Todo ou parte do nome do arquivo**, digite um nome de arquivo da tabela de informações de arquivo correta e clique em **Pesquisar**.
    4.  Na lista de arquivos, clique com o botão direito do mouse em um nome de arquivo da tabela de informações de arquivo correta e clique em **Propriedades**.
        **Observação** Dependendo da edição do sistema operacional, ou dos programas que são instalados em seu sistema, alguns arquivos que estão listados na tabela de informações de arquivo podem não estar instalados.
    5.  Na guia **Versão**, determine a versão do arquivo instalado no sistema, comparando-o com a versão documentada na tabela de informações de arquivos correta.
        **Observação** Atributos que não sejam a versão do arquivo podem ser alterados durante a instalação. A comparação de outros atributos de arquivo com as informações na tabela não é um método suportado para verificar se a atualização foi aplicada. Além disso, em alguns casos, arquivos podem ser renomeados durante a instalação. Se as informações sobre o arquivo ou a versão não estiverem presentes, use um dos outros métodos disponíveis para verificar a instalação da atualização.

-   **Verificação da Chave do Registro**

    Você também pode ser capaz de verificar os arquivos que esta atualização de segurança instalou examinando as chaves do Registro na **Tabela de referência** nesta seção.

    Essas chaves do Registro podem não conter uma lista completa dos arquivos instalados. Além disso, talvez essas chaves do Registro não sejam criadas corretamente se um administrador ou um OEM integrar ou adicionar a atualização de segurança aos arquivos fonte da instalação do Windows.

**Verificando a aplicação da atualização**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma atualização de segurança foi aplicada em um sistema afetado, é possível utilizar a ferramenta MBSA (Microsoft Baseline Security Analyzer). Consulte a seção **Orientação e ferramentas de detecção e implantação** neste boletim para obter mais informações.

-   **Verificação de Versão do Arquivo**

    Como existem várias versões do Microsoft Windows, as etapas a seguir podem ser diferentes no seu sistema. Se isso acontecer, consulte a documentação do produto para completar estas etapas.

    1.  Clique em **Iniciar** e, em seguida, clique em **Pesquisar**.
    2.  No painel **Resultados da Pesquisa**, clique em **Todos os arquivos e pastas**, em **Search Companion**.
    3.  Na caixa **Todo ou parte do nome do arquivo**, digite um nome de arquivo da tabela de informações de arquivo correta e clique em **Pesquisar**.
    4.  Na lista de arquivos, clique com o botão direito do mouse em um nome de arquivo da tabela de informações de arquivo correta e clique em **Propriedades**.
        **Observação** Dependendo da edição do sistema operacional, ou dos programas que são instalados em seu sistema, alguns arquivos que estão listados na tabela de informações de arquivo podem não estar instalados.
    5.  Na guia **Versão**, determine a versão do arquivo instalado no sistema, comparando-o com a versão documentada na tabela de informações de arquivos correta.
        **Observação** Atributos que não sejam a versão do arquivo podem ser alterados durante a instalação. A comparação de outros atributos de arquivo com as informações na tabela não é um método suportado para verificar se a atualização foi aplicada. Além disso, em alguns casos, arquivos podem ser renomeados durante a instalação. Se as informações sobre o arquivo ou a versão não estiverem presentes, use um dos outros métodos disponíveis para verificar a instalação da atualização.

-   **Verificação da Chave do Registro**

    Você também pode ser capaz de verificar os arquivos que esta atualização de segurança instalou examinando as chaves do Registro na **Tabela de referência** nesta seção.

    Essas chaves do Registro podem não conter uma lista completa dos arquivos instalados. Além disso, talvez essas chaves do Registro não sejam criadas corretamente se um administrador ou um OEM integrar ou adicionar a atualização de segurança aos arquivos fonte da instalação do Windows.

**Verificando a aplicação da atualização**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma atualização de segurança foi aplicada em um sistema afetado, é possível utilizar a ferramenta MBSA (Microsoft Baseline Security Analyzer). Consulte a seção Orientação e ferramentas de detecção e implantação neste boletim para obter mais informações.

-   **Verificação de Versão do Arquivo**

    Como existem várias versões do Microsoft Windows, as etapas a seguir podem ser diferentes no seu sistema. Se isso acontecer, consulte a documentação do produto para completar estas etapas.

    1.  Clique em **Iniciar** e, em seguida, clique em **Pesquisar**.
    2.  No painel **Resultados da Pesquisa**, clique em **Todos os arquivos e pastas**, em **Search Companion**.
    3.  Na caixa **Todo ou parte do nome do arquivo**, digite um nome de arquivo da tabela de informações de arquivo correta e clique em **Pesquisar**.
    4.  Na lista de arquivos, clique com o botão direito do mouse em um nome de arquivo da tabela de informações de arquivo correta e clique em **Propriedades**.
        **Observação** Dependendo da edição do sistema operacional, ou dos programas que são instalados em seu sistema, alguns arquivos que estão listados na tabela de informações de arquivo podem não estar instalados.
    5.  Na guia **Versão**, determine a versão do arquivo instalado no sistema, comparando-o com a versão documentada na tabela de informações de arquivos correta.
        **Observação** Atributos que não sejam a versão do arquivo podem ser alterados durante a instalação. A comparação de outros atributos de arquivo com as informações na tabela não é um método suportado para verificar se a atualização foi aplicada. Além disso, em alguns casos, arquivos podem ser renomeados durante a instalação. Se as informações sobre o arquivo ou a versão não estiverem presentes, use um dos outros métodos disponíveis para verificar a instalação da atualização.

-   **Verificação da Chave do Registro**

    Você também pode ser capaz de verificar os arquivos que esta atualização de segurança instalou examinando as chaves do Registro na **Tabela de referência** nesta seção.

    Essas chaves do Registro podem não conter uma lista completa dos arquivos instalados. Além disso, talvez essas chaves do Registro não sejam criadas corretamente se um administrador ou um OEM integrar ou adicionar a atualização de segurança aos arquivos fonte da instalação do Windows.

#### Informações de implantação
  
**Instalando a atualização**
  
Quando você instala esta atualização de segurança, o instalador verifica se alguns dos arquivos que estão sendo atualizados no sistema foram anteriormente atualizados por um hotfix da Microsoft.
  
Para obter mais informações sobre a terminologia que aparece neste boletim, como *hotfix*, consulte o [Artigo 824684 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/824684).
  
Esta atualização de segurança oferece suporte às seguintes opções de instalação:
  
| Opção             | Descrição                                                                                                                                               |  
|-------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------|  
| **/?, /h, /help** | Exibe ajuda sobre as opções compatíveis.                                                                                                                |  
| **/quiet**        | Suprime a exibição de status ou mensagens de erro.                                                                                                      |  
| **/norestart**    | Quando combinado com **/quiet**, o sistema não é reiniciado após a instalação, mesmo que uma reinicialização seja necessária à conclusão da instalação. |
  
**Observação** Para obter mais informações sobre o instalador wusa.exe, consulte o [Artigo 934307 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/934307).
  
**Verificando a aplicação da atualização**
  
-   **Microsoft Baseline Security Analyzer**
  
    Para verificar se uma atualização de segurança foi aplicada em um sistema afetado, é possível utilizar a ferramenta MBSA (Microsoft Baseline Security Analyzer). Consulte a seção **Orientação e ferramentas de detecção e implantação** neste boletim para obter mais informações.
  
-   **Verificação de Versão do Arquivo**
  
    Como existem várias versões do Microsoft Windows, as etapas a seguir podem ser diferentes no seu sistema. Se isso acontecer, consulte a documentação do produto para completar estas etapas.
  
    1.  Clique em **Iniciar** e insira um nome de arquivo de atualização em **Iniciar Pesquisa**.  
    2.  Quando o arquivo for exibido em **Programas**, clique com o botão direito do mouse no nome do arquivo e clique em **Propriedades**.  
    3.  Na guia **Geral**, compare o tamanho do arquivo com as tabelas de informações sobre arquivos fornecidas no artigo da KB do boletim.  
    4.  Você também pode clicar na guia **Detalhes** e comparar as informações, como versão do arquivo e data da última modificação, com as tabelas de informações de arquivos fornecidas no artigo da KB do boletim.  
    5.  Finalmente, você também pode clicar na guia **Versões Anteriores** e comparar as informações de arquivo da versão anterior do arquivo com as informações sobre a versão nova ou atualizada.
