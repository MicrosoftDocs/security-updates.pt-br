Security Bulletin

Microsoft Security Bulletin MS11-100 - Crítica
==============================================

Vulnerabilidades no .NET Framework podem permitir a elevação de privilégio (2638420)
------------------------------------------------------------------------------------

Publicado: quinta-feira, 29 de dezembro de 2011 | Atualizado: terça-feira, 10 de julho de 2012

**Versão:** 1.6

### Informações Gerais

#### Sinopse

Esta atualização de segurança elimina uma vulnerabilidade divulgada publicamente e três relatadas em particular no Microsoft .NET Framework. A mais severa dessas vulnerabilidades podia permitir a elevação de privilégio se um invasor não autenticado enviasse uma solicitação de Web especialmente criada ao site de destino. Um invasor que explorasse esta vulnerabilidade com êxito poderia tomar qualquer ação no contexto de uma conta existente no site de ASP.NET, inclusive executar comandos arbitrários. Para explorar esta vulnerabilidade, o invasor deve poder registrar uma conta no site de ASP.NET, e deve conhecer um nome de usuário existente.

Esta atualização de segurança é classificada Crítica para o Microsoft .NET Framework 1.1 Service Pack 1, Microsoft .NET Framework 2.0 Service Pack 2, Microsoft .NET Framework 3.5 Service Pack 1, Microsoft .NET Framework 3.5.1 e Microsoft .NET Framework 4 em todas as edições com suporte do Microsoft Windows. Para obter mais informações, consulte a subseção **Softwares afetados e não afetados**, nesta seção.

A atualização de segurança elimina as vulnerabilidades corrigindo a maneira como o .Net Framework lida com solicitações específicas e como o ASP .NET Framework autentica usuários e lida com conteúdo armazenado em cache. Para obter mais informação sobre as vulnerabilidades, consulte a subseção Perguntas frequentes relacionada à entrada da vulnerabilidade específica presente na próxima seção, **Informações sobre a vulnerabilidade**.

Esta atualização de segurança também elimina a vulnerabilidade descrita pela primeira vez no [Comunicado de Segurança da Microsoft 2659883](http://technet.microsoft.com/security/advisory/2659883).

**Recomendação.** A maioria dos clientes tem o recurso de atualizações automáticas habilitado e não precisará tomar nenhuma providência porque esta atualização de segurança será baixada e instalada automaticamente. Os clientes que não têm o recurso de atualizações automáticas habilitado precisarão verificar as atualizações e instalar esta atualização manualmente. Para obter informações sobre opções de configuração específicas em atualizações automáticas, consulte o [Artigo 294871 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/294871).

Para instalações feitas por administradores ou empresas ou por usuários finais que desejam instalar esta atualização de segurança manualmente, a Microsoft recomenda que os clientes apliquem a atualização de segurança usando software de gerenciamento de atualização ou verificando se há atualizações no serviço [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Consulte também a seção **Orientação e ferramentas de detecção e implantação** mais adiante neste boletim.

**Problemas conhecidos.** O [Artigo 2638420 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/2638420) documenta os problemas atualmente conhecidos que os clientes podem enfrentar ao instalar esta atualização de segurança. O artigo também documenta as soluções recomendadas para esses problemas.

#### Softwares afetados e não afetados

Os softwares a seguir foram testados para determinar quais versões ou edições foram afetadas. O ciclo de vida do suporte das outras versões ou edições já terminou ou elas não são afetadas. Para determinar o ciclo de vida do suporte para sua versão ou edição de software, visite o site [Ciclo de Vida do Suporte Microsoft.](http://go.microsoft.com/fwlink/?linkid=21742)

Perguntas frequentes relacionadas a esta atualização de segurança
-----------------------------------------------------------------

<span></span>
**Por que este boletim foi revisado em22 de maio de 2012?**
A Microsoft revisou este boletim para anunciar uma alteração de detecção referente ao KB2656352 do Microsoft .NET Framework 2.0 Service Pack 2 e corrigir um problema de instalação. Esta é apenas uma alteração de detecção. Não houve nenhuma alteração para os arquivos de atualização de segurança. Os clientes que já atualizaram seus sistemas com êxito não precisam fazer mais nada.

**Este boletim se refere ao MS12-035, Vulnerabilidade** **no .NET Framework pode permitir execução remota de código (2693777)?**
Sim. KB2656353, oferecido neste boletim, também aborda CVE-2012-0160 e CVE-2012-0161, que estão documentados no [MS12-035](http://technet.microsoft.com/pt-br/security/bulletin/ms12-035).

**O** **Microsoft .NET Framework 3.5 e o Microsoft .NET Framework 4.5 no Windows Developer Preview foram afetados por essa vulnerabilidade?**
Sim. A vulnerabilidade foi constatada depois do lançamento do Windows Developer Preview. Recomendamos que os clientes que estão usando o Developer Preview baixem e apliquem a atualização a seus sistemas. Para obter mais informações, inclusive links do Centro de Download da Microsoft, consulte os seguintes Artigos da Microsoft Knowledge Base:

| Softwares afetados                                        | Artigo da Microsoft Knowledge Base                                                                |
|-----------------------------------------------------------|---------------------------------------------------------------------------------------------------|
| Microsoft .NET Framework 3.5 no Windows Developer Preview | [Artigo 2656360 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/2656360) |
| Microsoft .NET Framework 4.5 no Windows Developer Preview | [Artigo 2656354 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/2656354) |

As atualizações de segurança estão disponíveis no [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) e no [](http://go.microsoft.com/fwlink/?linkid=21130)[Windows Update](http://go.microsoft.com/fwlink/?linkid=21130)</a>.

**Esta atualização contém alterações à funcionalidade relacionadas à segurança?**
Sim. Além das alterações listadas na seção Informações sobre a vulnerabilidade deste boletim, esta atualização inclui alterações funcionais no ASP.NET framework. A atualização limita a 1000 o número máximo de chaves de formulário, arquivos e membros JSON em uma solicitação. Esta alteração resulta em aplicativos ASP.NET, e solicitações com mais de 1000 desses elementos são rejeitadas. As solicitações desses tipos feitas pelos clientes serão negadas e uma mensagem de erro será exibida nos navegadores (geralmente um erro 500). O limite pode ser configurado de acordo com o aplicativo. Consulte o [Artigo 2661403 da Microsoft Knowledge Base](http://support.microsoft.com/kb/2661403).

A atualização também muda o formato dos tickets de autenticação de formulários de maneira que talvez seja incompatível com a versão mais antiga dos tickets de autenticação de formulários. Por exemplo, após a instalação da atualização em ambientes Web farm onde alguns servidores são atualizados e outros não, pode ser que os servidores atualizados gerem tickets de autenticação de formulários incompatíveis em servidores não atualizados. Consulte o [Artigo 2661404 da Microsoft Knowledge Base](http://support.microsoft.com/kb/2661404).

**Onde estão os detalhes das informações sobre o arquivo?** 
Consulte as tabelas de referência na seção **Implantação de atualização de segurança** para o local dos detalhes de informações sobre o arquivo.

**Como determino que versão do** **Microsoft .NET Framework está instalada?** 
Você pode instalar e executar várias versões do .Net Framework em um sistema, além de poder instalar as versões em qualquer ordem. Há várias maneiras de determinar quais versões do .Net Framework estão instaladas no momento. Para obter mais informações, consulte o [Artigo 318785 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/318785).

**Qual é a diferença entre** **o .NET Framework 4 e o .NET Framework 4 Client Profile?** 
Os pacotes redistribuíveis do .Net Framework versão 4 estão disponíveis em dois perfis: .NET Framework 4 e .NET Framework 4 Client Profile. O .NET Framework 4 Client Profile é um subconjunto do perfil do .NET Framework 4 que é otimizado para aplicativos clientes. Ele fornece funcionalidade para a maioria dos aplicativos clientes, incluindo recursos do Windows Presentation Foundation (WPF), Windows Forms, Windows Communication Foundation (WCF) e ClickOnce. Isso possibilita a implantação mais rápida e um pacote de instalação menor para aplicativos que têm como alvo o .NET Framework 4 Client Profile. Para obter mais informações, consulte o artigo de MSDN, [.NET Framework Client Profile](http://msdn.microsoft.com/en-us/library/cc656912.aspx).

**Existem duas atualizações listadas para a versão do** **Microsoft .NET Framework instalada em meu sistema. Devo instalar ambas as atualizações?** 
Sim. Os clientes devem aplicar todas as atualizações oferecidas para o software instalado em seus sistemas.

**Devo instalar estas atualizações de segurança em uma determinada sequência?** 
Não. Várias atualizações para uma versão do .NET Framework podem ser aplicadas em qualquer sequência. Recomendamos que várias atualizações para versões diferentes do .NET Framework sejam aplicadas em sequência, do número de versão mais baixo ao mais alto, mas essa sequência não é obrigatória.

**Eu tenho** **o .NET Framework 3.0 Service Pack 2 instalado; esta versão não está listada entre os softwares afetados neste boletim. Preciso instalar esta atualização?** 
Este boletim descreve uma vulnerabilidade na camada de recurso do .NET Framework 2.0 e no .NET Framework 4. O instalador do .NET Framework 3.0 Service Pack 2 é encadeado na configuração do .Net Framework 2.0 Service Pack 2, assim sendo, a instalação do anterior também instala o último. Portanto, os clientes que têm o .NET Framework 3.0 Service Pack 2 instalado precisam instalar as atualizações de segurança para o .NET Framework 2.0 Service Pack 2.

**Eu tenho** **o .NET Framework 3.5 Service Pack 1 instalado. Devo instalar alguma atualização adicional?** 
Este boletim descreve uma vulnerabilidade na camada de recurso do .NET Framework 2.0 e no .NET Framework 4. O instalador do .NET Framework 3.5 Service Pack 1 é encadeado tanto na configuração do .NET Framework 2.0 Service Pack 2 como no .NET Framework 3.0 Service Pack 2. Portanto, os clientes que têm o .NET Framework 3.5 Service Pack 1 instalado precisam instalar as atualizações de segurança para o .NET Framework 2.0 Service Pack 2.

**Uso uma versão mais antiga do software discutido neste boletim de segurança. O que devo fazer?** 
Os softwares afetados listados neste boletim foram testados para determinar que versões são afetadas. Outras versões passaram seu ciclo de vida de suporte. Para obter mais informações sobre o ciclo de vida do produto, visite o site [Ciclo de Vida do Suporte Microsoft](http://go.microsoft.com/fwlink/?linkid=21742).

Os clientes que possuem versões anteriores do software devem priorizar a migração para as versões com suporte, a fim de evitar uma possível exposição a vulnerabilidades. Para determinar o ciclo de vida do suporte para sua versão de software, consulte [Selecione um Produto para Obter Informações do Ciclo de Vida](http://go.microsoft.com/fwlink/?linkid=169555). Para obter mais informações sobre service packs para essas versões de software, consulte [Política de Suporte do Ciclo de Vida do Service Pack](http://go.microsoft.com/fwlink/?linkid=89213).

Os clientes que precisarem de suporte adicional para software mais antigo deverão entrar em contato com o representante da equipe de contas da Microsoft, o gerente técnico da conta ou o representante do parceiro Microsoft apropriado para obter opções de suporte personalizadas. Os clientes sem um contrato Alliance, Premier ou Authorized podem entrar em contato com o escritório de vendas local da Microsoft. Para obter informações de contato, visite o site de [Informações internacionais da Microsoft](http://go.microsoft.com/fwlink/?linkid=33329), selecione o país na lista de informações de contato e, em seguida, clique em **Ir** para ver uma lista de telefones. Ao ligar, peça para falar com o gerente de vendas local de Suporte Premier. Para obter mais informações, consulte as [Perguntas Frequentes sobre a Política do Ciclo de Vida do Suporte da Microsoft](http://go.microsoft.com/fwlink/?linkid=169557).

Colisões em HashTable que podem causar vulnerabilidade no DoS - CVE-2011-3414
-----------------------------------------------------------------------------

<span></span>
Uma vulnerabilidade de negação de serviço existe na maneira que o ASP.NET Framework manipula solicitações específicas, causando uma colisão de hash. Um invasor que explore com êxito esta vulnerabilidade pode enviar um pequeno número de solicitações específicas a um servidor de ASP.NET, fazendo com que o desempenho caia o suficiente para causar uma condição de negação de serviço.

Para exibir esta vulnerabilidade como uma entrada padrão na lista Common Vulnerabilities and Exposures, consulte [CVE-2011-3414](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-3414).

#### Fatores atenuantes para colisões em HashTable que podem causar vulnerabilidade no DoS - CVE-2011-3414

A atenuação refere-se a uma configuração, configuração comum ou prática recomendada geral, existindo num estado padrão, que possa reduzir a gravidade de exploração de uma vulnerabilidade. Os fatores de atenuação a seguir podem ser úteis em sua situação:

-   Por padrão, o IIS não está habilitado em nenhum sistema operacional Windows.
-   Os sites que rejeitam os tipos de conteúdo HTTP application/x-www-form-urlencoded ou multipart/form-data não estão vulneráveis
-   Por padrão, o ASP.NET não está instalado quando o .Net Framework estiver instalado. Somente os clientes que instalam e capacitam manualmente o ASP.NET estão provavelmente vulneráveis a este problema.

#### Soluções alternativas para colisões em HashTable que podem causar vulnerabilidade no DoS - CVE-2011-3414

A solução alternativa refere-se a uma configuração ou alteração de configuração que não corrige a vulnerabilidade subjacente mas que ajudaria a bloquear vetores de ataque conhecidos antes de aplicar a atualização. A Microsoft testou as soluções alternativas e estados a seguir como parte do cenário que implica se uma solução alternativa reduziria funcionalidade:

-   **Solução alternativa com base na configuração**

    A solução alternativa a seguir configura o limite do tamanho máximo de solicitação que o ASP.NET aceitará de um cliente. Diminuir o tamanho máximo de solicitação diminuirá a suscetibilidade do servidor ASP.NET a um ataque de negação de serviço.

    **Observação:** Faça o backup da configuração antes de fazer qualquer alteração.

    O valor desta configuração pode ser aplicado globalmente a todos os sites ASP.NET de um servidor adicionando a entrada à raiz web.config ou applicationhost.config. Ou pode-se restringir a configuração a um determinado site ou aplicativo adicionando-a a um arquivo web.config do site ou aplicativo específico. Para obter mais informações sobre a configuração ASP.NET, consulte o artigo da MSDN [Visão geral da configuração ASP.NET](http://msdn.microsoft.com/en-us/library/ms178683.aspx).

    1.  Se seu aplicativo usa ViewState, adicione a seguinte configuração no arquivo de configuração apropriado do ASP.NET para limitar o tamanho máximo de solicitação que ASP.NET aceitará de um cliente a 200 KB.`<configuration>  <system.web>  <httpRuntime maxRequestLength="200”/>  </system.web></configuration>`
    2.  Se seu aplicativo não usa ViewState, adicione a seguinte configuração no arquivo de configuração apropriado do ASP.NET para limitar o tamanho máximo de solicitação que ASP.NET aceitará de um cliente a 20 KB.`<configuration>  <system.web>  <httpRuntime maxRequestLength="20”/>  </system.web></configuration>`

    Para obter mais informações sobre a opção de configuração maxRequestLength, consulte a documentação da MSDN no [Elemento HTTP Runtime](http://msdn.microsoft.com/en-us/library/e1f13641.aspx).

    **Observação:** Aumentar a cadeia de caracteres HTTP padrão e solicitar limites de tamanho de cabeçalho aumentarão a suscetibilidade do servidor ao problema de negação de serviço descrito neste comunicado. Para obter mais informações sobre como configurar estes limites, consulte [configurações de registro de Http.sys para IIS](http://support.microsoft.com/kb/820129).

    **Impacto da solução alternativa:** As solicitações do cliente que excederem o maxRequestLength configurado resultarão em uma ConfigurationErrorsException em todo o lado do servidor, e um status de erro de http retornará ao cliente. Os limites configurados previamente serão usados como linha de base e poderão ser ajustados para evitar falhas do aplicativo. Configurar o valor de maxRequestLength com um valor menor também reduz a susceptibilidade do servidor a esta negação específica de serviço. Os aplicativos que aceitam uploads de arquivo de cliente e os que geram um viewstate grande são exemplos de aplicativos que podem ser impactados por esta solução alternativa.

    Depois que você aplica esta solução alternativa, a seguinte falha de solicitação de cliente pode aparecer em log de eventos do aplicativo.

    `Código o evento: ``3004 Mensagem`` do evento: O tamanho do lançamento excedeu os limites permitidos. Informação da exceção:``  ``   Tipo de exceção: HttpException     Mensagem da exceção: Tamanho máximo de solicitação excedido.   em System.Web.HttpRequest.GetEntireRawContent()   em System.Web.HttpRequest.FillInFormCollection()   em System.Web.HttpRequest.get_Form()   em System.Web.HttpRequest.get_HasForm()   em System.Web.UI.Page.GetCollectionBasedOnMethod(Boolean dontReturnNull)   em System.Web.UI.Page.DeterminePostBackMode()   em System.Web.UI.Page.ProcessRequestMain(Boolean includeStagesBeforeAsyncPoint, Boolean includeStagesAfterAsyncPoint)`

    Por outro lado, a falha de solicitação pode resultar na seguinte entrada num arquivo de log de IIS.

    `#Campos: s-ip cs-method cs-uri-stem cs-uri-query s-port cs-username c-ip ``cs(``User-Agent) sc-status sc-substatus sc-win32-status time-taken 127.0.0.1 POST /webSite/Default.aspx - 80 - 127.0.0.1 - 500 0 0 1268`

    **Remover a solução alternativa:** Para desfazer a solução alternativa, remova as seções de configuração que foram adicionadas à solução alternativa ou faça a restauração com um backup da configuração feito antes das soluções alternativas.

#### Perguntas frequentes das colisões em HashTable que podem causar vulnerabilidade no DoS - CVE-2011-3414

**Qual é o escopo da vulnerabilidade?** 
Essa é uma vulnerabilidade de negação de serviço (site em inglês). Essa vulnerabilidade torna viável o ataque de invasores não autenticados que degradam o desempenho de um site ASP.NET, criando uma condição de negação de serviço.

**O que provoca a vulnerabilidade?** 
A vulnerabilidade existe devido à maneira como os hashes ASP.NET especialmente criados solicitam e inserem esses dados em uma tabela de hash, causando uma colisão de hash. Quando muitas dessas colisões são encadeadas, o desempenho da tabela de hash fica muito ruim, o que leva à condição de negação de serviço.

**Para que um invasor pode usar a vulnerabilidade?** 
Um invasor pode usar essa vulnerabilidade para causar um ataque de negação de serviço e atrapalhar a disponibilidade de sites que usam ASP.NET.

**De que forma o invasor pode explorar a vulnerabilidade?** 
Um invasor não autenticado pode enviar um número pequeno de solicitações ASP.NET especialmente criadas para um site ASP.NET afetado, causando uma condição de negação de serviço.

**Quais são os principais sistemas que correm riscos com a vulnerabilidade?** 
Os sistemas que estão na Internet, com o ASP.NET instalado, estão principalmente correndo risco de apresentar essa vulnerabilidade. Além disso, os sites internos que usam a autenticação do ASP.NET Forms estão correndo risco de apresentar essa vulnerabilidade.

**O que a atualização faz?** 
A atualização aborda esta vulnerabilidade corrigindo como o .Net Framework manipula as solicitações específicas.

**Quando esse boletim de segurança foi lançado, essa vulnerabilidade já tinha sido divulgada publicamente?** 
Sim. Essa vulnerabilidade foi divulgada publicamente. Ela recebeu o número [CVE-2011-3414](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-3414) da lista Common Vulnerability and Exposure.

**Quando este boletim de segurança foi lançado, a Microsoft tinha recebido algum relatório informando que essa vulnerabilidade estava sendo explorada?** 
Não. A Microsoft não recebeu nenhuma informação indicando que essa vulnerabilidade tinha sido usada publicamente para atacar clientes quando este boletim de segurança foi lançado pela primeira vez.

**Como isto afeta a indústria?**
Os ataques a este tipo de vulnerabilidade são genericamente conhecidos como ataques de colisão de hash. Ataques como esses não são específicos às tecnologias da Microsoft e afetam outros fornecedores de software de serviço Web. Os clientes devem contatar seus fornecedores de plataforma Web para obter orientação ou atualizações.

O phishing (o "ph" é pronunciado com som de "f") online é uma maneira de descobrir informações pessoais ou financeiras dos usuários de computador, por meio de uma mensagem de email fraudulenta ou por meio de um site. Uma tentativa de phishing online comum inicia com uma mensagem de email que assemelha-se a um aviso oficial de uma fonte de confiança, tal como um banco, uma companhia de cartão de crédito, ou um comerciante on-line respeitável. Na mensagem de email, os destinatários são direcionados a um site fraudulento onde são solicitadas informações pessoais, tal como um número de conta ou senha. Esta informação é geralmente usada para roubo de identidade.

Esta vulnerabilidade é causada quando o .Net Framework verifica incorretamente o retorno das URLs durante o processo de autenticação de formulários.

A autenticação de formulários usa um ticket de autenticação que é criado quando um usuário faz logon em um site, e então controla o usuário por todo o site. O ticket de autenticação de formulários geralmente está contido dentro de um cookie. No entanto, o ASP.NET versão 2.0 oferece autenticação de formulários sem cookies, fazendo com o ticket passe por uma cadeia de caracteres de consulta. Em todos os casos, entretanto, o invasor não tem como forçar os usuários a exibir o conteúdo controlado pelo invasor. Em vez disso, um invasor teria que convencer os usuários a tomar medidas, geralmente clicando em um link numa mensagem de email ou numa mensagem do Instant Messenger que direciona os usuários ao site do invasor depois de fazer logon em um site esperado.

Os sistemas que estão na Internet, com o ASP.NET instalado, estão principalmente correndo risco de apresentar essa vulnerabilidade. Além disso, os sites internos que usam a autenticação do ASP.NET Forms estão correndo risco de apresentar essa vulnerabilidade.

**Se meu site usa uma web farm, eu deverei atualizar todos oscomputadores na web farm?** 
Sim. Esta atualização muda o formato do ticket de autenticação de formulário de tal maneira que fique incompatível com o formato anterior. Isso quer dizer que os tickets gerados com o novo comportamento não podem ser lidos por computadores com o comportamento antigo e vice versa. Os administradores cujos aplicativos usam autenticação baseada em formulários devem realizar etapas específicas ao implantar esta atualização de segurança, para assegurar que todos seus servidores mudem para o novo comportamento simultaneamente.

Para obter mais informações, consulte o [artigo do Technet da Microsoft 2659968](http://support.microsoft.com/kb/2659968).

**É possível atualizaros computadores na minha web farm um por vez, em vez de todos ao mesmo tempo?** 
Sim. Se não é viável implantar o MS11-100 a todos os servidores numa determinada web farm simultaneamente, você pode configurar uma opção de compatibilidade no seu arquivo web.config ou machine.config antes de instalar a atualização, para forçar o comportamento velho depois que a atualização for instalada. A opção de compatibilidade a ser usada é a seguinte:

`<``appSettings``>  <add key="aspnet:UseLegacyFormsAuthenticationTicketCompatibility" value="true" /></appSettings>`

Configurar esta opção permitirá que web farms com apenas algumas máquinas atualizadas continuem a funcionar corretamente. No entanto, é importante saber que os computadores com o conjunto de opção de configuração estarão num estado inseguro, e não irão se beneficiar da correção nesta atualização de segurança. Depois que o MS11-100 foi implementado a todos os computadores numa farm, a opção de configuração deve ser removida para que o novo comportamento de segurança seja capacitado.

**Isto irá atualizar a configuração** **ticketCompatibilityMode** **da autenticação de formulários do meu site?** 
Não. Desde que a correção nesta atualização de segurança altere o formato de tickets de autenticação de formulários, a configuração ticketCompatibilityMode não é mais suportada depois que o MS11-100 for instalado.

**O que a atualização faz?** 
A atualização elimina esta vulnerabilidade corrigindo a maneira que o ASP.NET Framework autentica os usuários.

Vulnerabilidade de armazenamento de ticket de autenticação do ASP.NET Forms - CVE-2011-3417
-------------------------------------------------------------------------------------------

<span></span>
Uma elevação de vulnerabilidade de privilégio existe na maneira em que o ASP.NET Framework manipula o conteúdo armazenado em cache quando a Autenticação de Formulários é usada com expiração móvel. Um invasor que explore com êxito esta vulnerabilidade poderia tomar qualquer ação, inclusive executar comandos arbitrários que o usuário podia realizar no site, no contexto do usuário visado. Em um cenário de ataque por email, o invasor pode explorar a vulnerabilidade enviando um link especialmente criado para o usuário e persuadindo-o a clicar no link. Não há como o invasor forçar os usuários a visitarem o site mal-intencionado. Em vez disso, um invasor teria que convencer os usuários a tomarem medidas, geralmente clicar em um link numa mensagem de e-mail ou em uma mensagem de Instant Messenger.

Para exibir esta vulnerabilidade como uma entrada padrão na lista Common Vulnerabilities and Exposures, consulte [CVE-2011-3417](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-3417).

#### Fatores atenuantes da Vulnerabilidade de armazenamento em cache de ticket de autenticação do ASP.NET Forms - CVE-2011-3417

A atenuação refere-se a uma configuração, configuração comum ou prática recomendada geral, existindo num estado padrão, que possa reduzir a gravidade de exploração de uma vulnerabilidade. Os fatores de atenuação a seguir podem ser úteis em sua situação:

-   Por padrão, as respostas do ASP.NET não são armazenadas no cache pelo OutputCache. O desenvolvedor do site tem que optar em armazenar no cache as saídas pela diretriz do OutputCache numa página.
-   O invasor que explorar com êxito essa vulnerabilidade poderá obter os mesmos direitos que o usuário visado. Os usuários cujas contas são configuradas com poucos direitos de usuário no sistema correm menos riscos do que aqueles que possuem direitos administrativos.
-   Por padrão, o IIS não estão instalado em nenhuma versão de sistema operacional afetada. Somente clientes que fazem a instalação manualmente isto são prováveis alvos deste problema.
-   Por padrão, o ASP.NET não está instalado quando o .NET estiver instalado. Somente os clientes que instalam e capacitam manualmente o ASP.NET estão provavelmente vulneráveis a este problema.

#### Soluções alternativas da Vulnerabilidade de armazenamento em cache de ticket de autenticação do ASP.NET Forms - CVE-2011-3417

A solução alternativa refere-se a uma configuração ou alteração de configuração que não corrige a vulnerabilidade subjacente mas que ajudaria a bloquear vetores de ataque conhecidos antes de aplicar a atualização. A Microsoft testou as soluções alternativas e estados a seguir como parte do cenário que implica se uma solução alternativa reduziria funcionalidade:

-   **Restrinja os cookies de autenticação de formulários para canais SSL**

    Abra o aplicativo ou arquivo global web.config depois de fazer uma cópia de backup.

    Configure requireSSL =" true" no elemento &lt;forms&gt;, conforme mostrado no seguinte código.

    `<forms loginUrl="Secure\Login.aspx"    requireSSL="true" />`

    **Como desfazer a solução alternativa.**

    Restaure seu arquivo web.config da sua cópia de backup.

-   **Desabilite expiração móvel para cookies de autenticação de formulários**

    1.  Abra o aplicativo ou arquivo global web.config depois de fazer uma cópia de backup
    2.  Configure slidingExpiration="false" no elemento &lt;forms&gt;, conforme mostrado no código a seguir.`<forms slidingExpiration="false"/>`

    **Impacto da solução alternativa.** Os usuários seriam forçados a fazer o logon novamente depois que seus tickets expirarem.

    **Como desfazer a solução alternativa.**

    Restaure seu arquivo web.config da sua cópia de backup.

-   **Desabilite o OutputCache**

    1.  Faça backup de sua pasta de aplicativo web.
    2.  Remova todas as diretrizes &lt;%@ OutputCache %&gt; das suas páginas web.
    3.  Se seu site é um aplicativo MVC, também remova todos os atributos de OutputCache de suas ações de controlador.

    **Impacto da solução alternativa.** Os usuários seriam forçados a fazer o logon novamente depois que seus tickets expirarem.

    **Como desfazer a solução alternativa.**

    Restaure sua pasta de aplicativos Web.

#### Perguntas frequentes da Vulnerabilidade de armazenamento em cache de ticket de autenticação do ASP.NET Forms - CVE-2011-3417

**Qual é o escopo da vulnerabilidade?** 
Isto é uma elevação de vulnerabilidade de privilégio onde um invasor obtém acesso remoto não autenticado a outra conta no aplicativo do .NET.

**O que provoca a vulnerabilidade?** 
Esta vulnerabilidade é causada quando o .Net Framework manipula incorretamente o conteúdo armazenado em cache quando a Autenticação de Formulários é usada com expiração móvel.

**Para que um invasor pode usar a vulnerabilidade?** 
Um invasor que explore com êxito esta vulnerabilidade poderia tomar qualquer ação, inclusive executar comandos arbitrários que o usuário podia realizar no site, no contexto do usuário visado.

**De que forma o invasor pode explorar a vulnerabilidade?** 
Em um cenário de ataque por email, o invasor pode explorar a vulnerabilidade enviando um link especialmente criado para o usuário e persuadindo-o a clicar no link. Não há como o invasor forçar os usuários a visitarem o site mal-intencionado. Em vez disso, um invasor teria que convencer os usuários a tomarem medidas, geralmente clicar em um link numa mensagem de e-mail ou em uma mensagem de Instant Messenger.

**O que é Autenticação de Formulários no** **ASP.NET?** 
A autenticação de formulários usa um ticket de autenticação que é criado quando um usuário faz logon em um site, e então controla o usuário por todo o site. O ticket de autenticação de formulários geralmente está contido dentro de um cookie. No entanto, o ASP.NET versão 2.0 oferece autenticação de formulários sem cookies, fazendo com o ticket passe por uma cadeia de caracteres de consulta. Para obter mais informações, consulte o artigo (em inglês) do MSDN, [Explained: Forms Authentication in ASP.NET 2.0](http://msdn.microsoft.com/en-us/library/ff647070.aspx).

**Quais são os principais sistemas que correm riscos com a vulnerabilidade?** 
Os sistemas que estão na Internet, com o ASP.NET instalado, estão principalmente correndo risco de apresentar essa vulnerabilidade. Além disso, os sites internos que usam a autenticação do ASP.NET Forms estão correndo risco de apresentar essa vulnerabilidade.

**O que a atualização faz?** 
A atualização modifica a maneira que o ASP.NET manipula conteúdo armazenado em cache.

#### Informações de implantação

**Instalando a atualização**

Quando você instala esta atualização de segurança, o instalador verifica se alguns dos arquivos que estão sendo atualizados no sistema foram anteriormente atualizados por um hotfix da Microsoft.

Se você já tiver instalado um hotfix para atualizar um desses arquivos, o instalador copiará os arquivos RTMQFE, SP1QFE ou SP2QFE para o sistema. Do contrário, o instalador copiará os arquivos RTMGDR, SP1GDR ou SP2GDR para o sistema. As atualizações de segurança podem não conter todas as variações desses arquivos. Para obter mais informações sobre esse comportamento, consulte o [Artigo 824994 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/824994).

Para obter mais informações sobre o instalador, consulte o [Artigo 832475 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/832475).

Para obter mais informações sobre a terminologia que aparece neste boletim, como *hotfix*, consulte o [Artigo 824684 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/824684).

**Verificando a aplicação da atualização**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma atualização de segurança foi aplicada em um sistema afetado, é possível utilizar a ferramenta MBSA (Microsoft Baseline Security Analyzer). Consulte a seção **Orientação e ferramentas de detecção e implantação** neste boletim para obter mais informações.

-   **Verificação de versão de arquivo**

    Como existem várias versões do Microsoft Windows, as etapas a seguir podem ser diferentes no seu sistema. Se isso acontecer, consulte a documentação do produto para completar estas etapas.

    1.  Clique em **Iniciar** e, em seguida, clique em **Pesquisar**.
    2.  No painel **Resultados da Pesquisa**, clique em **Todos os arquivos e pastas**, em **Search Companion**.
    3.  Na caixa **Todo ou parte do nome do** **arquivo** , digite um nome de arquivo da tabela de informações de arquivo correta e clique em **Pesquisar**.
    4.  Na lista de arquivos, clique com o botão direito do mouse em um nome de arquivo da tabela de informações de arquivo correta e clique em **Propriedades**.
        **Observação** Dependendo da edição do sistema operacional ou dos programas instalados em seu sistema, alguns arquivos listados na tabela de informações de arquivos podem não estar instalados.
    5.  Na guia **Versão**, determine a versão do arquivo instalado no sistema, comparando-o com a versão documentada na tabela de informações de arquivos correta.
        **Observação** Atributos que não sejam a versão do arquivo podem ser alterados durante a instalação. A comparação de outros atributos de arquivo com as informações na tabela não é um método suportado para verificar se a atualização foi aplicada. Além disso, em alguns casos, arquivos podem ser renomeados durante a instalação. Se as informações sobre o arquivo ou a versão não estiverem presentes, use um dos outros métodos disponíveis para verificar a instalação da atualização.

    Você também pode ser capaz de verificar os arquivos que esta atualização de segurança instalou examinando as chaves do Registro na **Tabela de referência** nesta seção.

    Essas chaves do Registro podem não conter uma lista completa dos arquivos instalados. Além disso, talvez essas chaves do Registro não sejam criadas corretamente se um administrador ou um OEM integrar ou adicionar a atualização de segurança aos arquivos fonte da instalação do Windows.

#### Informações de implantação

**Instalando a atualização**

Quando você instala esta atualização de segurança, o instalador verifica se alguns dos arquivos que estão sendo atualizados no sistema foram anteriormente atualizados por um hotfix da Microsoft.

Se você já tiver instalado um hotfix para atualizar um desses arquivos, o instalador copiará os arquivos RTMQFE, SP1QFE ou SP2QFE para o sistema. Do contrário, o instalador copiará os arquivos RTMGDR, SP1GDR ou SP2GDR para o sistema. As atualizações de segurança podem não conter todas as variações desses arquivos. Para obter mais informações sobre esse comportamento, consulte o [Artigo 824994 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/824994).

Para obter mais informações sobre o instalador, consulte o [Artigo 832475 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/832475).

Para obter mais informações sobre a terminologia que aparece neste boletim, como *hotfix*, consulte o [Artigo 824684 (em inglês) da Microsoft Knowledge Base](http://support.microsoft.com/kb/824684).

**Verificando a aplicação da atualização**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma atualização de segurança foi aplicada em um sistema afetado, é possível utilizar a ferramenta MBSA (Microsoft Baseline Security Analyzer). Consulte a seção **Orientação e ferramentas de detecção e implantação** neste boletim para obter mais informações.

-   **Verificação de versão de arquivo**

    Como existem várias versões do Microsoft Windows, as etapas a seguir podem ser diferentes no seu sistema. Se isso acontecer, consulte a documentação do produto para completar estas etapas.

    1.  Clique em **Iniciar** e, em seguida, clique em **Pesquisar**.
    2.  No painel **Resultados da Pesquisa**, clique em **Todos os arquivos e pastas**, em **Search Companion**.
    3.  Na caixa **Todo ou parte do nome do** **arquivo** , digite um nome de arquivo da tabela de informações de arquivo correta e clique em **Pesquisar**.
    4.  Na lista de arquivos, clique com o botão direito do mouse em um nome de arquivo da tabela de informações de arquivo correta e clique em **Propriedades**.
        **Observação** Dependendo da edição do sistema operacional ou dos programas instalados em seu sistema, alguns arquivos listados na tabela de informações de arquivos podem não estar instalados.
    5.  Na guia **Versão**, determine a versão do arquivo instalado no sistema, comparando-o com a versão documentada na tabela de informações de arquivos correta.
        **Observação** Atributos que não sejam a versão do arquivo podem ser alterados durante a instalação. A comparação de outros atributos de arquivo com as informações na tabela não é um método suportado para verificar se a atualização foi aplicada. Além disso, em alguns casos, arquivos podem ser renomeados durante a instalação. Se as informações sobre o arquivo ou a versão não estiverem presentes, use um dos outros métodos disponíveis para verificar a instalação da atualização.

-   **Verificação da chave do Registro**

    Você também pode ser capaz de verificar os arquivos que esta atualização de segurança instalou examinando as chaves do Registro na **Tabela de referência** nesta seção.

    Essas chaves do Registro podem não conter uma lista completa dos arquivos instalados. Além disso, talvez essas chaves do Registro não sejam criadas corretamente se um administrador ou um OEM integrar ou adicionar a atualização de segurança aos arquivos fonte da instalação do Windows.

**Verificando a aplicação da atualização**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma atualização de segurança foi aplicada em um sistema afetado, é possível utilizar a ferramenta MBSA (Microsoft Baseline Security Analyzer). Consulte a seção **Orientação e ferramentas de detecção e implantação** neste boletim para obter mais informações.

-   **Verificação de versão de arquivo**

    Como existem várias versões do Microsoft Windows, as etapas a seguir podem ser diferentes no seu sistema. Se isso acontecer, consulte a documentação do produto para completar estas etapas.

    1.  Clique em **Iniciar** e insira um nome de arquivo de atualização na caixa **Iniciar Pesquisa**.
    2.  Quando o arquivo for exibido em **Programas**, clique com o botão direito do mouse no nome do arquivo e clique em **Propriedades**.
    3.  Na guia **Geral**, compare o tamanho do arquivo com as tabelas de informações sobre arquivos fornecidas no artigo da KB do boletim.
        **Observação** Dependendo da edição do sistema operacional ou dos programas instalados em seu sistema, alguns arquivos listados na tabela de informações de arquivos podem não estar instalados.
    4.  Você também pode clicar na guia **Detalhes** e comparar as informações, como versão do arquivo e data da última modificação, com as tabelas de informações de arquivos fornecidas no artigo da KB do boletim.
        **Observação** Atributos que não tenham a versão do arquivo podem ser alterados durante a instalação. A comparação de outros atributos de arquivo com as informações na tabela não é um método suportado para verificar se a atualização foi aplicada. Além disso, em alguns casos, arquivos podem ser renomeados durante a instalação. Se as informações sobre o arquivo ou a versão não estiverem presentes, use um dos outros métodos disponíveis para verificar a instalação da atualização.
    5.  Finalmente, você também pode clicar na guia **Versões Anteriores** e comparar as informações de arquivo da versão anterior do arquivo com as informações sobre a versão nova ou atualizada.

**Verificando a aplicação da atualização**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma atualização de segurança foi aplicada em um sistema afetado, é possível utilizar a ferramenta MBSA (Microsoft Baseline Security Analyzer). Consulte a seção **Orientação e ferramentas de detecção e implantação** neste boletim para obter mais informações.

-   **Verificação de versão de arquivo**

    Como existem várias versões do Microsoft Windows, as etapas a seguir podem ser diferentes no seu sistema. Se isso acontecer, consulte a documentação do produto para completar estas etapas.

    1.  Clique em **Iniciar** e insira um nome de arquivo de atualização na caixa **Iniciar Pesquisa**.
    2.  Quando o arquivo for exibido em **Programas**, clique com o botão direito do mouse no nome do arquivo e clique em **Propriedades**.
    3.  Na guia **Geral**, compare o tamanho do arquivo com as tabelas de informações sobre arquivos fornecidas no artigo da KB do boletim.
        **Observação** Dependendo da edição do sistema operacional ou dos programas instalados em seu sistema, alguns arquivos listados na tabela de informações de arquivos podem não estar instalados.
    4.  Você também pode clicar na guia **Detalhes** e comparar as informações, como versão do arquivo e data da última modificação, com as tabelas de informações de arquivos fornecidas no artigo da KB do boletim.
        **Observação** Atributos que não tenham a versão do arquivo podem ser alterados durante a instalação. A comparação de outros atributos de arquivo com as informações na tabela não é um método suportado para verificar se a atualização foi aplicada. Além disso, em alguns casos, arquivos podem ser renomeados durante a instalação. Se as informações sobre o arquivo ou a versão não estiverem presentes, use um dos outros métodos disponíveis para verificar a instalação da atualização.
    5.  Finalmente, você também pode clicar na guia **Versões Anteriores** e comparar as informações de arquivo da versão anterior do arquivo com as informações sobre a versão nova ou atualizada.
